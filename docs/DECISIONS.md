# DECISIONS

Architecture decision records. Format per entry:

```
## ADR-NNN: Title
- Status: accepted | superseded by ADR-MMM
- Date: YYYY-MM-DD
- Decision: one sentence.
- Rationale: short paragraph.
```

New ADRs append to the end. Changing a CONTRACT.md invariant requires an ADR here and, once the runtime exists, a decision object in the graph.

## ADR-001: Five object types

- Status: accepted
- Date: 2026-06-10
- Decision: The schema is exactly mission, branch, artifact, evidence, decision; adding a type is a gated decision.
- Rationale: Schema growth is itself an experiment. Starting minimal forces every proposed type to argue for itself with evidence from real use, and the gate makes that argument a recorded event rather than a drive-by commit.

## ADR-002: One-bit authority

- Status: accepted
- Date: 2026-06-10
- Decision: Each capability is either auto or gated; publishing and self-modification are always gated.
- Rationale: Maturity ladders (trust levels, graduated autonomy tiers) are bureaucracy the lab can evolve later if evidence demands it. One bit is auditable at a glance and cheap to flip per capability via a gated decision.

## ADR-003: Lab is a layered pack on core

- Status: accepted; supersedes the "exactly five object types" invariant of ADR-001
- Date: 2026-06-10
- Decision: The lab is a layered pack on the activegraph-packs core pack. It adds exactly THREE object types (mission, branch, decision) and reuses core primitives: core artifact for outputs, core observation + evaluation for evidence, core task for work dispatch, core source for ingested pages/repos/papers.
- Rationale: Core is the lingua franca; a parallel ontology would violate the conventions the lab is built on.
- New invariant: adding a fourth lab object type requires a gated decision object AND an ADR.

## ADR-004: No lab_interface pack

- Status: accepted
- Date: 2026-06-10
- Decision: The upstream communication + chat packs provide threads, messages, intents, and message injection. Thread = branch is one relation (discusses: thread → branch) plus one answer behavior in the lab pack. The lab_kernel/lab_interface split collapses into a single lab pack (`lab_pack/`).
- Rationale: The interface already exists upstream; we add a projection, not a layer.

## ADR-005: Standalone repo, packs as pinned git dependency

- Status: accepted
- Date: 2026-06-10
- Decision: activegraph-lab consumes activegraph-packs as a git dependency (`activegraph-packs @ git+https://github.com/yoheinakajima/activegraph-packs`, pinned to a commit SHA), with a single-line editable-install override for local dev. The lab is the first external consumer of the packs conventions; friction encountered while consuming them is evidence — record it as observations under the mission, and propose upstream issues as artifacts, never as direct edits to the packs repo. Bumping the pinned SHA is a gated decision the lab records about itself.
- Rationale: The packs repo is a reference library, not a product; the lab is a product.

## ADR-006: Workers via emergent coordination

- Status: accepted
- Date: 2026-06-10
- Decision: The lab never calls domain packs directly. Its work behavior writes core task objects with a routing convention (task.kind or tags); research and codebase pack behaviors react. No adapters/ directory.
- Rationale: Packs compose through graph state, not function calls — the packs repo's central design rule. An adapter layer would be a coordinator in disguise.

## ADR-007: 'paused' is a branch status

- Status: accepted
- Date: 2026-06-10
- Decision: The branch status enum gains `paused` (proposed|scoped|active|paused|interpreting|decided|archived), replacing the scoped + metadata.paused workaround for the pause steering verb.
- Rationale: Pause is an owner-visible state the feed must render and steering must round-trip (pause → resume). Overloading `scoped` made the projection lie about intent. Enum values on a lab-owned type are a code change plus this ADR — not a new object type, so no gated decision is required (ADR-003 gates types, not fields).

## ADR-008: Relation call-convention handling

- Status: accepted
- Date: 2026-06-10
- Decision: The lab writes relations in signature order — `add_relation(source_id, target_id, type)` — everywhere, and reads mixed graphs through one documented helper, `lab_pack/compat.py:decode_relation`, which discriminates per relation (object ids contain `#`, relation type names never do). All lab code that reads relations goes through this helper.
- Rationale: The packs repo is split on argument order (core/research/tool_gateway type-first; chat signature-order); a composed graph holds both encodings. Signature order is what runtime view traversal and relation queries require, so the lab writes it; the decode shim is quarantined in one place, linked to the friction observation seeded under the mission, and goes away if upstream standardizes (the lab's draft issue artifact proposes exactly that).

## ADR-009: Storage — native PostgresEventStore, selected at boot

- Status: accepted
- Date: 2026-06-11
- Decision: The event store is activegraph's native persistence layer (ag-coder pattern: a dedicated `activegraph` Postgres schema, framework-owned tables, fork/replay native, via `pip install "activegraph[postgres]"`). `DATABASE_URL` present → Postgres; absent → SQLite under `data/` (the dev/fixture default; fixtures stay keyless and deterministic). Backend selection lives in exactly one place (`lab_pack/storage.py`, kernel); no other code may know which store is active, and all projections read through runtime/event APIs, never raw SQL against framework tables.
- Rationale: The log is the source of truth; the store is the framework's concern, not the lab's. One selection point keeps the swap auditable and the rest of the codebase backend-blind.
- Note (2026-06-11): selection now reads `LAB_DATABASE_URL` first, falling back to `DATABASE_URL`, then SQLite. Replit reserves `DATABASE_URL` when its managed-Postgres module is present, colliding with explicit secrets at publish time; the lab sidesteps the reserved name rather than fighting the platform, and the fallback preserves every existing environment. `LAB_DATABASE_URL` is a credential exactly like `DATABASE_URL` (ADR-011); the sentinel audit covers both.
- Note (2026-06-11, connection resilience): PostgresEventStore holds a single boot-lifetime connection; serverless Postgres (Neon) suspends idle computes and kills it. Observed twice in production with the identical signature — first write after an idle suspend fails `AdminShutdown` ("terminating connection due to administrator command"), every subsequent write fails `OperationalError` ("the connection is closed") until restart; ADR-023 surfaced both correctly, nothing committed. `storage.harden_store` (armed at server boot, idempotent) wraps the store's operations: on a connection-class error (psycopg `OperationalError` — which covers AdminShutdown and SSL EOF/reset — or `InterfaceError`, plus the raw `ssl.SSLError`/`ConnectionError` escape shapes) it re-establishes the connection and retries the operation exactly once; a second failure, or a failed reconnect, surfaces structured per ADR-023. Non-connection errors are never retried — a retried append whose first attempt actually committed must surface as `UniqueViolation`, not duplicate (`UNIQUE(id, run_id)` makes the retry double-commit-safe). Each reconnect records a `store_reconnected` entry on the diagnostics ring buffer, never the event log (the log may be the casualty — ADR-023); long-idle appends get a cheap `SELECT 1` probe first, and a failed reconnect leaves the old connection in place so the next operation self-heals. Touching the store's connection internals stays legal here and only here; the upstream candidate (reconnect-with-bounded-retry belongs in the store) is queued in LIVE_FINDINGS.
- OPEN: memory_gateway's own store stays local-SQLite for now.

## ADR-010: Deployment — Replit, continuously running

- Status: accepted
- Date: 2026-06-11
- Decision: The lab deploys to Replit as a continuously running server; all durable state lives in managed Postgres (ADR-009), never the filesystem. A single operator authenticates with a bearer token (`LAB_OPERATOR_TOKEN`); all projections are publicly readable.
- Rationale: One operator, one always-on process, zero filesystem state makes redeploys idempotent: the schema either has events (resume) or it doesn't (seed).

## ADR-011: Public log policy

- Status: accepted
- Date: 2026-06-11
- Decision: Operator chat is part of the public log, deliberately. No secrets in any event payload, observation, artifact, boot log, or error path — enforced by an automated sentinel audit (DATABASE_URL is a credential too).
- Rationale: The lab's pitch is an inspectable agent; a private side-channel would undercut it. The cost is discipline about payloads, which the audit makes mechanical instead of aspirational.
- Note (2026-06-11, ADR-023 amendment to error hygiene): an exception's CLASS and a sanitized one-line MESSAGE are not secrets — withholding them turned a one-line diagnosis into three sessions of log forensics. Error responses and the `/lab/errors` projection carry both. Payload contents, env values, URLs/DSNs (credentials ride in them), and filesystem paths remain secrets: `_sanitize_error_text` redacts them mechanically and the sentinel audit greps the projection like every other public surface. The OAuth endpoints keep their fixed bodies — an unauthenticated password-guessing surface gets no oracle.

## ADR-012: Code residency — the four-tier ladder

- Status: accepted
- Date: 2026-06-11
- Decision: Code lives in one of four tiers. KERNEL (GitHub only, forever): the gate behavior, auth middleware, event loop/runtime wiring, replay machinery, the storage adapter, and the seam/code loaders themselves — the thing that governs self-modification is never subject to it. SEAMS (graph-stored, gated): prompts, feed narration templates, whitelisted behavior settings values. GRAPH CODE (graph-stored, gated, dark by default): behaviors and tools drafted as artifacts, sandbox-tested, promoted only through an approved decision AND `LAB_ALLOW_GRAPH_CODE=1`. PLUMBING (GitHub, pragmatically): server, UI, scripts.
- Enforcement: `lab_pack/kernel.py` is the manifest of protected module paths; the seam and code loaders refuse any graph artifact that names, imports from, shadows, or monkeypatches a manifest entry. The manifest itself is kernel. The seam-eligible settings whitelist is kernel.
- Rationale: Graph-stored seams and code give perfect replay provenance — the code that ran is in the log that replays. The kernel stays in git for bootstrap and security. Self-modification is a capability like any other: one bit, gated, absolute (ADR-002).

## ADR-013: Public site structure — the blog is the front door

- Status: accepted
- Date: 2026-06-10
- Decision: The root of the lab's domain is a blog: published posts rendered server-side from the graph (artifacts with status=published), newest first. Each post page renders the post plus its provenance subgraph: originating branch, evidence observations, chat messages on that branch, the publish decision, and prior draft versions. The notebook feed moves to /lab and shows everything — all branches (active, proposed, decided, archived), all chats, all decisions including resolved ones.
- Rationale: The work is the product; the blog is the front door, the notebook is the open workshop behind it. Rendering posts from the graph (not the drafts/ mirror) keeps the UI a pure projection of the log, and "Show the work" makes the inspectability pitch concrete on every post.
- Note: publishing now appends an `artifact.published` marker event through the gate path. Marker events (here and ADR-015) carry no graph-state projection — they are log entries the lab's own projections read — so the "lab adds no event types" line in ARCHITECTURE.md is amended to "no state-bearing event types".

## ADR-014: Editorial policy — notes accumulate, research is earned

- Status: accepted
- Date: 2026-06-10
- Decision: Artifacts of kind=blog_draft gain a post_kind field: note | research | build. Small single-finding posts are notes; multi-evidence or multi-branch syntheses are research; posts about constructing the lab itself are build. Notes do not auto-draft one-per-finding: findings accumulate as queued observations, and a digest behavior proposes one combined note when unpublished queued findings reach setting.digest_min_findings (default 3). Research/build drafts require a decided branch with at least setting.research_min_evidence (default 3) linked evidence objects, or a synthesis of ≥2 decided branches whose combined evidence meets the same bar. When pending publish decisions reach setting.max_drafts_pending (default 5), automatic drafting idles and records an observation — the operator's attention is also a budget. The operator can always request a draft on a branch via chat; an explicit request is spent attention, so it bypasses the pending cap.
- Rationale: Fire-per-finding floods the inbox and produces thin posts. Accumulation plus thresholds makes volume a policy, and every threshold is a seam setting — tuning editorial policy is self-modification through the gate, not a redeploy.

## ADR-015: Operator controls — pause and a daily cost ceiling

- Status: accepted
- Date: 2026-06-10
- Decision: A global pause and a daily cost ceiling, both persisted as events and rebuilt from the log at boot (restart-proof, the same mechanism as the daily call cap). POST /lab/pause and /lab/resume append lab.paused / lab.resumed marker events; mutations are operator-token-gated like everything else. While paused, every LLM behavior except answer idles, recording one behavior-skipped observation per behavior per pause episode (not per event); answer stays live — the operator can always talk to the lab, and answer is cheap and budget-counted. The cost ceiling uses activegraph's native cost accounting (cost_usd on llm.responded events), capped by setting.daily_cost_cap_usd (default 5.00, seam-whitelisted); blocked-by-cost attempts log like blocked-by-count.
- Rationale: The kill switch and the spend limit must survive restarts, which means they live in the log, not in process memory or env vars. Keeping answer alive while paused preserves the one channel the operator steers through.
- Note (2026-06-11): a production process that inherited paused=true from the log booted with a dead worker — the resumed-boot path only drained the runtime when findings were backfilled, so the backlog that replay requeues (every event after the log's last runtime.idle) sat parked, and the operator's lab.resumed (an event nothing subscribes to) woke no run cycle (evt_1702/1845/1846/1847 vs evt_1590/1616, diagnosed over MCP). Amendment, locked by the paused_boot fixture: pause state rebuilds from the log BEFORE the first drain; the boot run cycle always happens; resume drains immediately. Paused gates which behaviors fire (everything but answer idles), never whether the worker runs.

## ADR-016: MCP surface — external assistants read the lab, the operator's authority stays human

- Status: accepted
- Date: 2026-06-10
- Decision: The lab exposes an MCP server (streamable HTTP) at `/mcp` on the existing lab server, so external AI assistants can read the lab and, with operator authority, talk to it. Tools come in two tiers. READ (no special authority — this data is already public via the blog, /lab, and /trace): `get_status` (the healthz projection), `get_feed` (paginated), `get_branch` (full timeline for a branch id, chats interleaved), `get_pending_decisions` (with evidence summaries), `get_post` (published post + provenance), `list_posts`, `list_seams`. OPERATOR (requires authority): `send_chat` — message a branch as the operator; the message lands in the public log like any operator message (ADR-011). EXCLUDED BY DESIGN: approve/reject decisions, pause/resume, and seam promotion are deliberately NOT tools. The inbox is the one place only the human operator exists; routing approval through an external AI would reduce the gate's guarantee to "an AI approved it." This is a recorded non-capability, not an omission.
- Access: the `/mcp` endpoint requires `Authorization: Bearer LAB_MCP_TOKEN` — a NEW secret, distinct from `LAB_OPERATOR_TOKEN`, revocable independently. It never grants inbox or pause authority even though it permits `send_chat`: `LAB_MCP_TOKEN` is refused on `/lab/decision` and `/lab/pause`, and `LAB_OPERATOR_TOKEN` is refused on `/mcp` (strict two-way separation; covered by a cross-authority test). Constant-time compare and 401/403 semantics match the rest of the server; `/mcp` mutations share the existing in-memory rate limiter. MCP-originated chats are tagged `source=operator_via_mcp` in the event payload so the public log distinguishes the human from their assistant.
- Implementation: the streamable-HTTP protocol (JSON-RPC 2.0 over POST, single-JSON responses — a mode the spec permits) is implemented minimally by hand in `server/mcp.py`. The official Python MCP SDK was evaluated and declined: its streamable-HTTP transport is ASGI-only, and the lab server is stdlib `http.server` (kernel wiring, ADR-012) — adopting the SDK would mean adding a Starlette/uvicorn stack and restructuring kernel server wiring for one endpoint. No new dependency is taken; if the server ever moves to ASGI, swapping in the SDK is the natural follow-up. No new state: every tool is a projection of the event log or a call into the existing chat path. The auth check for `/mcp` lives in `server/lab_server.py` (already in the kernel manifest); the manifest itself is not modified — graph code still cannot reach the token because `os.environ` is manifest-protected.
- Rationale: the lab's pitch is an inspectable agent; an MCP surface makes the inspection programmable for the operator's own assistants without widening the gate. Splitting the token keeps "my assistant can read and chat" revocable without touching the operator's own authority.
- Amendment (2026-06-10): `/mcp/<token>` (path segment) is accepted as an alternate presentation of the same credential — constant-time-compared against `LAB_MCP_TOKEN`, identical authority to the header path (`send_chat` yes; decisions/pause still refused), wrong path token → 401. Reason: claude.ai's custom-connector UI cannot send a static bearer header (its credential fields are OAuth-only), which made header-only `/mcp` unusable from the Claude clients; header auth stays for everything else. Security note: the URL is now a credential — treat connector URLs like tokens (they can land in client configs, browser history, and intermediary logs). On the lab's side the token is never echoed or stored: access logging is disabled, error bodies carry only fixed messages, and the sentinel audit exercises the URL-token path. Rotation = rotate `LAB_MCP_TOKEN`; both presentations rotate together.
- Note (2026-06-11): send_chat's bounded reply wait is `setting.mcp_reply_wait_seconds` (default 15, seam-whitelisted in lab_pack/kernel.py). The original fixed 60s wait exceeded claude.ai's own tool timeout, so the client errored before the structured reply_pending partial could be returned — the bound must come in under the client's, and tuning it is self-modification through the gate. (SUPERSEDED by ADR-034: send_chat is now commit-and-return-immediately, so there is no bounded wait at all; the setting is retired in place.)
- OPEN: OAuth 2.1 with Dynamic Client Registration is the eventual proper implementation for the Claude clients; the URL token is the pragmatic bridge until then. (Resolved by ADR-017.)

## ADR-017: OAuth 2.1 + DCR on the MCP surface — stateless, single-operator, keyed from LAB_MCP_TOKEN

- Status: accepted
- Date: 2026-06-11
- Amends: ADR-016 (resolves its OPEN item; everything else in ADR-016 — the tool tiers, the excluded-by-design gate authority, the two-way token separation — stands unchanged).
- Decision: the MCP surface gains a single-operator OAuth 2.1 authorization server so claude.ai's web connector (whose credential fields are OAuth-only) can complete its handshake against a plain `/mcp` URL. Design principle: STATELESS — no token table, no client table, nothing secret ever written to the event log or any storage. Every credential the server issues (client_id, authorization code, access/refresh token) is an HMAC-SHA256-signed payload (type, client_id, scope, expiry), keyed from `LAB_MCP_TOKEN` via a fixed derivation, and verified later by recomputing the signature. Rotating `LAB_MCP_TOKEN` therefore revokes every client registration, code, and token at once — the same revocation story as the legacy presentations, which REMAIN: bearer-header `/mcp` and path-token `/mcp/<token>` keep working with identical authority (`send_chat` yes; decisions/pause/seams still excluded).
- Endpoints (all on the existing stdlib `http.server`, no ASGI, no new dependency — the ADR-016 reasoning about the MCP SDK still applies): `GET /.well-known/oauth-authorization-server` and `GET /.well-known/oauth-protected-resource` (RFC 8414 / RFC 9728 metadata, also served with the `/mcp` path suffix; PKCE S256 required, `token_endpoint_auth_methods_supported: ["none"]` — public clients, no client_secret); `POST /register` (RFC 7591 DCR: returns a deterministic client_id, the HMAC of the sorted redirect_uris — idempotent re-registration with no registry; redirect_uris must be https claude.ai/claude.com callbacks or localhost, anything else is refused); `GET /authorize` (a minimal HTML page with one password field — the OPERATOR pastes `LAB_MCP_TOKEN`; correct token, constant-time-compared → 302 to the redirect_uri with a 60-second HMAC-signed code binding client_id + redirect_uri + PKCE challenge; wrong token → 401 with a fixed body, no oracle detail); `POST /token` (code + PKCE verifier → 24h access token + 30d refresh token; refresh_token grant rotates the pair; fixed RFC 6749 error bodies, constant-time comparisons throughout).
- `/mcp` 401s carry `WWW-Authenticate: Bearer resource_metadata="…/.well-known/oauth-protected-resource/mcp"` per the MCP auth spec, so clients discover the flow. Bearer routing on `/mcp`: the legacy `LAB_MCP_TOKEN` compare runs first; a credential shaped like a signed blob then takes OAuth verification (failure → 401 `invalid_token` so clients refresh); any other mismatch keeps the legacy 403.
- Hard rule (sentinel-enforced): no token, code, or `LAB_MCP_TOKEN`-derived value appears in any event payload, log line, or error body. The 302 Location and the `/token` response body are the intended delivery channels and the only places a credential appears; `scripts/test_public_safety.py` runs the full flow end to end and audits the minted code, both tokens, their signature halves, and the derived signing key against the whole public corpus.
- Implementation: the protocol lives in `server/oauth.py`, which holds NO secret — every function takes the signing key as an explicit argument, derived from `LAB_MCP_TOKEN` only inside `server/lab_server.py` (kernel, ADR-012). Importing `server.oauth` therefore yields inert functions: graph code cannot reach the key (`os.environ` and `server.lab_server` are manifest-protected), so the kernel manifest itself is again not modified — the same posture ADR-016 took with `server/mcp.py`. The `/register`, `/token`, and `/authorize` POSTs share the existing in-memory mutation rate limiter (they are unauthenticated endpoints; the authorize form is the password-guessing surface).
- Accepted trade: statelessness means an authorization code cannot be marked used — replay inside its 60-second window with the same PKCE verifier mints a token. The code only ever transits the operator-initiated redirect to an allowlisted claude.ai/claude.com/localhost callback, the verifier never leaves the client, and the alternative is a token table — the thing this design exists to avoid. Likewise individual tokens cannot be revoked: revocation is rotation, which is already the lab's story.
- Rationale: the URL-token bridge made the connector URL itself a credential; OAuth removes the secret from the URL entirely (the operator authenticates once on the authorize page) without adding storage, a framework, or any new authority. One secret still rules the surface, and the human gate (ADR-016's excluded-by-design list) is untouched.

## ADR-018: The charter is a seam — operator-authored, versioned, injected verbatim

- Status: accepted
- Date: 2026-06-11
- Decision: the lab gains an operator-authored mission charter as seam `charter.mission` — versioned, gated, hot-loaded, replay-recorded, and injected VERBATIM (a delimited CHARTER block) into the context assembly of exactly three behaviors: plan, interpret, and draft_writer. v1 ships as the FILE DEFAULT (`lab_pack/prompts/charter.md` — operator-authored content, committed by the operator's own build pipeline), so unlike every other seam the file default counts as version 1, not 0: the first graph-stored proposal is v2, parented on the file's v1, and it activates only through the self_modify gate like any seam. `charter.mission` is the only whitelisted charter surface; the kernel-manifest body scan applies to charter bodies exactly as to prompts.
- Replay fidelity: the three behaviors stamp `charter.mission` alongside their prompt version in `seam_versions` on every output, so the log records which charter was in force for each decision the model made — and since replay never re-fires behaviors, those records replay verbatim (the same mechanism as prompt seams, ADR-012 Phase 4).
- Exclusions: answer does NOT receive the charter — it reports graph state to the operator and must not restate portfolio priorities as if they were findings. The charter composes WITH prompt seams (description = active prompt body + active charter block), so promoting either surface independently recomposes the live context without restart.
- Mechanical note: the upstream prompt loader (`load_prompts_from_dir`) refuses any `prompts/*.md` without TOML frontmatter, so charter.md carries the standard two-line frontmatter; the BODY — the thing injected, resolved, and versioned — is exactly the operator's charter text. The charter prompt name is excluded from provider-side behavior identification (its body appears inside three behaviors' contexts).
- Rationale: the lab's portfolio judgment (what to verify, build, measure; how much introspection is too much; what voice is honest) is policy, not plumbing — it belongs in an inspectable, versioned, gated surface the lab can eventually propose changes to itself, not scattered through prompt prose the operator would have to diff by hand.

## ADR-019: Per-behavior model routing as seam settings; budget defaults raised behind a kernel ceiling

- Status: accepted
- Date: 2026-06-11
- Decision: each lab llm_behavior's model is a seam setting — `setting.model.plan`, `setting.model.interpret`, `setting.model.draft_writer`, `setting.model.answer`, plus `setting.model.default` for behaviors without their own entry (all seam-whitelisted; dotted seam names map to underscored LabSettings fields). Defaults: plan/interpret/draft_writer = `claude-opus-4-8` (the deliberate, expensive plane); answer/default = `claude-sonnet-4-20250514` (the fast plane). The resolution is stamped onto `behavior.model` at boot (after `apply_approved`) and on every `setting.model.*` hot-load, so a seam override reroutes without restart — and because the runtime reads `behavior.model` at prompt assembly and records it on every `llm.requested` event, the per-behavior resolution is in the log natively, with no lab-side event plumbing. A routed model the active provider does not recognize is skipped (the provider default stays) rather than raised: cross-provider misconfiguration must not kill the worker.
- Cost accounting: stays the runtime's native per-model reporting — `cost_usd` on `llm.responded`, priced per model family by the provider — never a hardcoded single price. The daily budget rebuild (ADR-015) is unchanged and therefore correct across mixed-model days by construction.
- Budget defaults: `daily_cost_cap_usd` default rises 5.00 → 50.00 (opus-priced deliberate work needs headroom). NEW kernel constant `ABSOLUTE_DAILY_COST_CEILING_USD = 100.00` (`lab_pack/kernel.py`): enforced at the budget path's single enforcement point (`effective_cost_cap`), NOT seam-modifiable, NOT MCP-modifiable — every cap mechanism (settings default, approved `setting.daily_cost_cap_usd` seam, the MCP `set_budget` control of ADR-021) clamps to it. Raising the ceiling is a git change, never a graph one.
- Found while wiring this: the pack loader registers FRESH canonical-named copies of every behavior (`lab.plan`, …) — deliberately, upstream — so the existing prompt-seam hot-load, which mutated only the module originals, never reached the live runtime's registration (the seams fixture asserted the original's attribute, not the registered copy's). Fixed here for prompts, the charter, and model routing alike: `bind_live_behaviors` captures the runtime's copies at every build/resume, and all seam mutation paths write through `behaviors_named` to both objects; the fixtures now assert against `rt.get_behavior("lab.<name>")`.
- Rationale: which model thinks for which behavior is policy the lab should be able to argue about and change through the gate — but spend needs a floor the graph cannot move, or a single approved seam (or compromised assistant) could turn the routing table into a blank check.

## ADR-020: Lab-local research worker — plumbing that fills the dispatch gap, droppable by design

- Status: accepted
- Date: 2026-06-11
- Decision: the lab gains a local research worker (`lab_pack/research_worker.py`, PLUMBING tier) implementing the EXISTING routing contract (ADR-006): it reacts to core task objects routed `research.deep_research` exactly as the work behavior already emits them. Dispatch code is untouched; the worker is droppable — disabled (or deleted), the capability-gap path takes over unchanged, and the day an upstream pack becomes task-reactive the module goes away.
- Shape: two reactive stages over the one log. `research_intake` (plain behavior) claims a routed task — a `research_progress` observation plus an `executes` relation, with a registry entry the dispatch gap check consults (BehaviorGraph cannot iterate relations) — and proposes fetch `capability_call`s through tool_gateway ONLY (CONTRACT), capped per task by `setting.research_fetch_cap` (seam, default 8); each returning `tool_result` is a progress patch on the task (worker contract: progress ≤60s — every external call is bounded well under it). When the last fetch lands: one synthesis-request observation, or — when every fetch failed — the task patched failed with the error IN the event (errors propagate, never a silent abort). `research_worker` (llm_behavior, model `setting.model.research_worker`, default the plan tier `claude-opus-4-8`) synthesizes with per-claim source attribution: findings citing fetched URLs only (unattributable findings are dropped and counted — attribution is the contract, not a suggestion), written as core observations with `source_ids`, plus one evaluation linked `supported_by` to the branch; then the task completes and the EXISTING work→interpret outcome path takes it from there. Inert LLM output (budget, pause, parse) fails the task with the reason recorded.
- Enablement: `research_worker_enabled` defaults False in the pack — no embedding, fixture, or test reaches the network by surprise — and the server boot enables it; the live lab always runs the worker. Budgets and pause ride the provider wrapper like every lab llm_behavior. Claimed/synthesized task ids rebuild from their observations on resume; in-flight fetches do not survive a restart — the stall watchdog releases the task (accepted: a restart costs one task retry, not a coordinator).
- Rationale: the capability gap was the honest state of the evidence base, but honesty about not working is worth less than working honestly — a worker that lives in the lab, obeys the same gateway/budget/pause rails, and disappears the moment upstream fills the gap keeps the emergent-coordination claim intact while making research branches actually execute.

## ADR-023: Chat path failure domain — the append is the only fail point; everything else degrades, diagnosably

- Status: accepted
- Date: 2026-06-11
- Decision: in any chat path (POST /chat and MCP send_chat), the comm_message append is the ONLY step whose failure may fail the request — and even that failure is structured (exception class + sanitized one-line message, per the ADR-011 amendment noted there), never a generic 500. Everything post-commit is individually guarded: thread/relation upkeep, registry-cache updates, save, response assembly, and the reply drain. A post-commit failure records a sanitized entry on the diagnostics ring buffer plus a best-effort `chat_path_degraded` observation, and the response is `status=reply_pending` carrying the committed message ids — NEVER an error after a successful append. The reply is always triggered on the runtime worker once the append succeeded: a bounded-wait timeout leaves the job queued, a degraded path submits it fire-and-forget, and a client disconnect is irrelevant to its completion. Thread resolution no longer trusts the in-process discusses cache alone: on a miss it reads the log's relations (ADR-008 decode) before creating a new thread — the cache is a cache, the log is the truth.
- Leaf cause (the incident this hardens against): a pg_dump/pg_restore'd lineage left the `events.seq` BIGSERIAL sequence behind the restored rows. Boot appends landed in the leading seq gap and succeeded; once nextval reached the restored block, EVERY durable append raised `psycopg.errors.UniqueViolation (events_pkey)` from `store.append` inside `graph.emit` — AFTER the event entered the in-memory log. Projections (including MCP forensics) showed the message as committed while durability had silently stopped, the request 500'd generically, and the reply drain never started. SQLite cannot reproduce it: AUTOINCREMENT derives the next rowid from the table; BIGSERIAL trusts nextval. Fix: `lab_pack/storage.repair_sequences` realigns the sequence past max(seq) at resumed boot, before the runtime opens the store. The storage adapter is the one module allowed to touch a framework table (ADR-009 already makes it the only backend-aware code); projections still never read framework tables.
- Diagnostics: a volatile in-process ring buffer (last 50 entries: ts, class, sanitized message, request kind, related event ids) served read-only at `GET /lab/errors` and the MCP READ tool `get_errors` (added to the ADR-016 READ tier; it grants nothing the stderr log didn't already hold, minus the secrets). Deliberately NOT log-derived and reachable before readiness: its job is to stay readable when appending to the log — or booting the runtime — is the thing that failed. This is a scoped exception to the "all user-visible state derives from the event log" invariant, recorded in CONTRACT.md: memory only, lost on restart, never authoritative.
- Rationale: the incident took three sessions to corner because the failure was post-commit, the error was generic, and the only divergence signal (in-memory log vs durable rows) had no projection. Splitting the failure domain makes the committed/not-committed boundary explicit in every response; the ring buffer plus structured errors turn the next one into a three-minute read.

## ADR-024: Bind first — readiness is a phase, not a precondition

- Status: accepted
- Date: 2026-06-11
- Decision: the server binds its socket BEFORE building the runtime; the boot (replay, registry rebuild, drain) runs on the runtime worker behind it. `/healthz` answers from bind time with `ready=false` and the boot phase (`starting → loading → draining → ready | failed`); `/lab/errors` is likewise never gated (ADR-023). Every other request gets `503 + Retry-After: 5` with the phase in the body until the drain completes; a failed boot reports `phase=failed` with the sanitized boot error on `/healthz` and keeps refusing with 503.

Health checks need the socket, not the graph — and an honest "starting, retry in 5" beats a dead port. ## ADR-025: Truthful steering — post-mutation replies, the activate verb, branch-keyed decisions, the MCP side door closed

- Status: accepted
- Date: 2026-06-12
- Incident: the operator sent "Activate this branch" to branch#62 over MCP (evt_3676) and the answer behavior replied "Activating this branch now... task dispatched" — but no activate verb existed, branch#62 stayed proposed, no task was created, no worker ran. The model's reply is generated BEFORE the handler runs, so it narrated an action it could neither perform nor verify, and the prompt ("acknowledge the action briefly") invited exactly that. Separately, chat `approve` keyed pending decisions by subject_ref while the thread keys by branch, so approving a publish decision (subject = the artifact) from chat was a silent no-op.
- Decision (a), truthful replies — the principle, not just a verb: steering mutations apply FIRST, and for any action-shaped message the reply is composed deterministically from POST-mutation graph state. Every applied verb appends a `lab.steering_applied` marker event (the marker family, ADR-013/015/021) recording verb, branch, message, and mutation refs; the reply cites that event id — a reply may claim only actions whose mutation event it cites. A verb that applies nothing (already paused, nothing pending) says so. An action request no verb supports draws an explicit refusal naming the supported verb set. The model's pre-mutation narration is used only for questions about state; its prompt now forbids action claims outright. Verb matching is word-boundary ("activate" no longer hides inside "deactivate"/"reactivate", "pause" inside "unpause").
- Decision (b), activate/deactivate: steering gains `activate` (proposed/scoped → active) and `deactivate` (active → proposed) for the thread's own branch. Operator authority, MCP-ALLOWED — the ADR-021 reversibility argument applies exactly: a wrong activation costs one task retry and is undone by its twin verb, categorically unlike a promotion. Activation records the operator's rationale as a `branch_activated` observation linked supported_by to the branch, then patches status — the EXISTING work dispatch reacts to the patch; nothing new dispatches. (`recrawl` joins the same reversible tier: a crawl_request source scoped to the mission target_url that ingest already reacts to, plus a fresh in-process crawl episode — replay never re-fires behaviors, so a resumed lab whose crawl stalled needs the nudge. Registry state resets; the log is append-only and existing observations are never touched.)
- Decision (c), decision keying, closed correctly: pending decisions index by BRANCH as well as subject_ref — promote decisions via their subject, publish/self_modify via metadata.lab_branch_id (rebuilt from the graph on resume like every registry). Chat approve/reject with exactly one pending applies it; multiple pending lists the ids and mutates nothing; zero is an honest no-op. AND: messages tagged source=operator_via_mcp are REFUSED for approve/reject with the reason in the reply — the inbox is human-only (ADR-016/021). The keying bug was an accidental side door that happened to fail closed; this fixes it closed, not open: fixing the lookup without the source check would have silently granted MCP the decision authority ADR-016 explicitly excludes.
- Rationale: the lab's pitch is an inspectable agent, and an agent that narrates fictional actions in its own public log is the opposite of inspectable. Tying every action claim to a citable mutation event makes the chat surface exactly as auditable as the rest of the log — and the refusal path turns "the verb doesn't exist" from a hallucination prompt into recorded evidence of what operators actually ask for.

## ADR-026: Decision resolution carries rationale — the registry learns reasons, not bits; MCP may annotate, never resolve

- Status: accepted
- Date: 2026-06-12
- Gap (production evidence): decision objects carried only the PROPOSER's rationale; the operator's approve/reject was a bare status flip, so every rejection reason of the past week existed only outside the graph. The rejected-decision registry — which informs seam proposals and drafting — therefore learned from bits, not reasons.
- Decision (a), resolution rationale: `POST /lab/decision` accepts an optional `rationale` string; the resolution patch is ONE event recording it as `metadata.resolution_rationale` with `metadata.resolved_by=operator` — the proposer's `rationale` field stays untouched (the pitch and the verdict are different evidence). Absent means absent: no placeholder is recorded. The UI's approve/reject buttons open an optional text field (prefilled empty; skippable) and the confirm posts it. The rejected-decision registry stores and exposes `resolution_rationale` (rebuilt from decision metadata on resume, like every registry), so seam proposals, draft_writer evidence (`_item_context` on cited decisions), and interpret cite the OPERATOR's reasons, not just the proposer's pitch. Chat approve/reject (ADR-025, human-only) resolves through the same path, recording the operator's message as the rationale — the reason was already in the log; now the resolution event cites it too. The feed narrates resolutions from the operator's reason when one exists.
- Decision (b), annotation: new MCP operator-tier tool `annotate_decision(decision_id, note)` attaches a public, `operator_via_mcp`-attributed annotation to a PENDING decision — pre-review notes, recommendations, draft rationale. The note is an observation (`metadata.lab=decision_annotation`) appended to the decision's `metadata.annotation_refs`; the handler can only create the observation and append the ref — no code path touches status, and an already-resolved decision draws a tool error. It does NOT and cannot resolve. When the operator later resolves in the UI, pending annotations are linked into the resolution's evidence (`evidence_refs`) and the rationale field is prefilled from the most recent annotation (operator can edit before submitting). approve/reject remain EXCLUDED from MCP; annotation is commentary, not authority.
- Decision (c), backfill: nothing — the log is append-only; the registry carries rationales going forward. Old resolutions stay bare, honestly.
- Fixtures/tests: `decision_rationale` fixture (resolve-with-rationale lands on the event + registry + resume rebuild; annotate → pending decision shows it → resolution links it and the inbox projection exposes the prefill source IN FULL; annotate cannot change status); `scripts/test_mcp.py` covers the tool end to end and re-asserts `LAB_MCP_TOKEN` is refused on `/lab/decision`; the sentinel audit (`scripts/test_public_safety.py`) covers the new tool's output like every public surface; `check_ui` drives the two-step resolve form and the annotation prefill in jsdom.
- Rationale: the gate's value is not the bit, it is the judgment behind the bit — and a judgment that never enters the graph cannot teach the lab anything. Recording the operator's reasons on the resolution event makes rejections evidence; letting the operator's assistant stage commentary (but never authority) keeps the inbox human-only while making the human's eventual decision better-argued and cheaper to record.

## ADR-027: Rejection is teaching, not burial — rejected promotes land on decided, direction becomes evidence, activation dispatches it

- Status: accepted
- Date: 2026-06-13
- Incident (production): the operator rejected promote decision#266 with a `resolution_rationale` carrying explicit continuation direction — primary sources, named repos and papers, a clear promotion bar. The gate's reject path auto-archived branch#62 (evt_13850); archived branches were not steerable (activate refused anything but proposed/scoped), so the direction sat recorded in the graph and unreachable by any execution path. The FIRST operator attempt to teach through the gate (the whole point of ADR-026) buried the student with the lesson attached.
- Decision (a), rejected promote → decided: the gate's reject path no longer archives. A rejected promote patches the branch to `decided` — the same terminal-but-steerable state as approval — and, when the resolution carries a `resolution_rationale`, records it VERBATIM as an `operator_direction` observation linked `supported_by` to the branch: the direction is branch evidence, visible in the branch view. Consequences accepted: rejection no longer frees a `max_open_branches` slot (decided branches never did on approval either — symmetric now), and NO code path archives a branch anymore; `archived` remains a legal status for the legacy population, and resurrection (b) is its exit.
- Decision (b), activate widens: the `activate` steering verb now applies to `decided` AND `archived` branches alongside proposed/scoped. decided → active is a continuation; archived → active is a deliberate operator resurrection and is recorded as such (the `branch_activated` observation carries `resurrected: true` and says so; the `lab.steering_applied` event records `previous_status`). A deliberate activation is a fresh dispatch episode: the work behavior's dispatch dedup resets for that branch (the recrawl move — registries are caches, the log is append-only), so a NEW task dispatches. Archived branches are chat-able for exactly one steering verb — activate; every other verb or action request draws an honest refusal naming it, and plain questions get the model's answer prefixed with a deterministic archived notice.
- Decision (c), direction dispatch: a branch carrying operator_direction observations dispatches the LATEST direction verbatim on the task (`task.metadata.operator_direction`), and the activation message rides on the branch (`metadata.activation_message`) into the task too — URLs the operator mentions when activating steer the worker's sources (the existing mentioning-links principle, extended to the verb: decision#266's direction named its sources without schemes, so the resurrection message must be able to supply fetchable ones). The research worker reads the direction three ways: source URLs in the direction join the fetch candidates FIRST (the operator named them deliberately — ahead of the claim/mission defaults under the fetch cap), the synthesis request carries the direction whole in metadata AND as a delimited OPERATOR DIRECTION block in its text (the seam-truncation lesson: an excerpt in the request IS a truncation in the output), and so the worker's LLM context contains what the operator ordered — fixture-locked with a recording mock provider. The `_OPERATOR_DIRECTIONS` registry rebuilds from the observations on resume like every registry — and ALSO from rejected promote decisions whose metadata carries a `resolution_rationale` with no direction observation (the legacy decision#266 shape): no backfill events are appended (ADR-026's stance stands), the cache simply learns to read what is already in the log, so branch#62's recorded teaching is reachable the moment this deploys.
- Fixture note: `branch_lifecycle`'s old `reject → archived` assertion ENCODED the bug — it locked the burial in as correct behavior. Updated deliberately to `reject → decided` plus the direction observation; the new `rejection_lifecycle` fixture drives the full arc (reject-with-direction → decided + evidence → chat reactivation cited via steering_applied → fresh task carrying the direction → worker fetches the direction's named source, direction mock-asserted in the synthesis context → a new promote decision surfaces) and the resurrection/refusal semantics for archived branches.
- Rationale: ADR-026 made the operator's reasons evidence; this ADR makes them ACTIONABLE. A rejection that archives is a verdict; a rejection that re-arms the branch with the operator's direction is a feedback loop — and the loop's last mile (direction → dispatched context → worker's eyes) is exactly the part that silently didn't exist.

## ADR-028: The note/comment steering verb — recording operator commentary that is not a command

- Status: accepted
- Date: 2026-06-13
- Incident (production): an operator erratum sent to branch#62 (evt_17441) was REFUSED — truthful steering (ADR-025) correctly refuses to invent actions for an action-shaped message no verb supports, but there was no way to RECORD operator commentary that isn't a command. The operator's correction had nowhere to land in the graph.
- Decision: steering gains a `note` verb (alias `comment`, word-boundary matched per ADR-025). It records the operator's FULL message as a branch observation (`metadata.lab=operator_note`, `metadata.kind=operator_note`, linked `supported_by` to the branch — the same shape as `branch_activated`/`operator_direction`), appends a `lab.steering_applied` marker, and the reply confirms with the recorded event id (`Applied: operator note recorded as observation … — recorded at evt_…`). It claims NOTHING else: status is never touched, no work dispatches. Operator authority, MCP-allowed — commentary is not control over the branch, so the ADR-021 reversibility line does not even apply (there is nothing to reverse); it rides through the existing chat/send_chat path and is tagged with the message's source like every other steering verb. The verb is checked AFTER every command verb so a real command ("pause this branch — just a note to self") still applies the command; `note` is the fallback for commentary that is not a command. An archived branch (ADR-027) accepts `note` alongside `activate` — recording a comment changes nothing, so it is safe in any state.
- Fixture: `truthful_steering` Phase 4 — note over `operator_via_mcp` records an `operator_note` observation with the full message + an honest confirmation citing the steering event, branch status is unchanged, and a command carrying the word "note" still applies the command.
- Rationale: ADR-025's refusal path turned "the verb doesn't exist" from a hallucination into recorded evidence of what operators ask for; this closes the other half — when the operator's message is commentary rather than a command, the lab records it faithfully instead of refusing. The inbox stays human-only (no decision authority is added); `note` is the lowest-authority verb there is.
- Amendment (2026-06-14, Phase 4): the `note` verb's intent — free-text commentary that needs no pending decision — is now also reachable as a BARE MCP tool, `annotate_branch(branch_id, note)` (operator-tier, same authority as send_chat/annotate_decision). It records the same `operator_note` observation linked `supported_by` to the branch and changes NO status, sharing one recording shape with the steering verb via `lab_pack/tools.py:annotate_branch_fn`. Unlike the send_chat `note` verb, it needs no command wording and does not route through the answer behavior — a deterministic place for an erratum or aside to land (evt_17441). It adds no new authority (annotation is the lowest-authority operation; ADR-026's `annotate_decision` already established public, attributed, non-resolving MCP commentary) and the inbox stays human-only. Fixture: `scripts/test_mcp.py` — annotate_branch records an attributed operator_note observation carrying the full text, leaves status unchanged, is readable in the branch timeline, and errors on a bogus id; the sentinel audit covers its output like every public surface.

## ADR-029: Provenance truth — the footer records the real per-behavior model, and a brief-governed post's evidence is what it footnotes

- Status: accepted
- Date: 2026-06-13
- Gap (external review): the blog provenance footer printed one model (`model claude-opus-4-8`) for every post, but the pipeline is per-role (ADR-019: draft_writer/plan on the deliberate plane, answer/default on the fast plane). The single footer flattened the routing into one name — it rounded the split up. Two adjacent provenance gaps rode along: the claims-coverage check fired on the framing/intro paragraph in ~100% of posts (and was routinely overridden via the escape hatch, training the guardrail into noise), and a draft produced under an operator brief recorded `evidence_refs` from the queued-findings pile while the body actually footnoted different entities (artifact#868/#882 listed digest findings the body never cited).
- Decision (a), per-behavior footer: the footer records the model ACTUALLY used by each behavior that produced the artifact — draft_writer always, plus every contributing LLM behavior found in the request's per-item provenance (`findings_context.created_by` → interpret/plan/research_worker/…). Each model is the value the runtime stamps on that behavior's `llm.responded`/`llm.requested` events (ADR-019); the lab reads it from the session's `model_by_behavior` cache, which the provider wrapper records from the same `model` kwarg the runtime passes — so the footer is `model draft_writer=…, interpret=…, research_worker=…`, the real split, never a constant. A behavior with no recorded call (a fixture on the raw mock) falls back to `last_model`.
- Decision (b), brief-governed evidence: when a draft was produced under an operator brief (ADR-014 escape hatch), the decision's `evidence_refs`, the artifact's `observation_ids`, and the provenance block all derive from the entities the BODY footnotes (read from footnote definition lines), not the queued findings the request carried as available evidence. With no brief, the existing evidence_refs path is unchanged.
- Decision (c), coverage exempts framing: the claims-coverage check exempts a pure framing/intro OPENING paragraph — the first substantive paragraph, if it makes no specific factual claim (no digit, percentage, comparative/quantitative term, code identifier (snake_case/camelCase), object-id reference, or URL — heuristic justified in `_makes_factual_claim`). A substantive opener (numeric or named) still flags, and a substantive claim anywhere but the opening paragraph still flags. A FIRED check now means a real unfootnoted substantive claim, not boilerplate.
- Fixtures: `provenance_footer` (a draft_writer on opus citing research_worker on sonnet footers BOTH, per role — the split, not one name); `draft_writer` Phase 4 (framing intro exempt; a numeric opener and a camelCase-named opener both flag); `editorial` Phase 5 (a brief-governed draft's `evidence_refs`/`observation_ids`/provenance block match the footnoted ids, not the queued pile).
- Rationale: the lab's pitch is an inspectable agent, and a footer that names one model for a multi-model pipeline is the inspectable surface lying by rounding. Tying the footer to the recorded per-behavior model, the evidence to what the body cites, and the guardrail to real claims makes every provenance line on a post true rather than approximately true.

## ADR-031: Capability self-check — a behavior consults the tool set before asserting absence

- Status: accepted
- Date: 2026-06-14
- Incident (production): branch#847 was activated to fetch and verify activegraph's replay implementation files. The router tagged it `codebase.code_task` (the pre-fix keyword router sent anything naming a repo there), the research_worker only reacts to `research.deep_research`, so no pack reacted and the dispatch gap check fired. - Decision: "no pack reacted to this routing" is a ROUTING fact, NEVER proof the lab cannot do the work. A task whose real lane's capability is genuinely absent still records an honest `capability_gap` — the worker disabled, or a `codebase.code_task` write with no reactor, is still an honest gap.
- Fixture: `capability_self_check` — a research intent forced into the codebase lane (worker live) records a `routing_miss` naming `codebase.code_task → research.deep_research` and NO gap; a genuine `codebase.code_task` write task still records an honest `capability_gap`. The pre-existing `capability_gap` and `research_worker` fixtures lock the honest-gap path unchanged.
- Rationale: this is an honesty invariant in the family of ADR-026's "the registry learns reasons, not bits" and the voice policy — the lab must not slander its own capabilities. A gap is evidence; a FALSE gap is misinformation that compounds (it spawned branch#911 — see ADR-032). Grounding the assertion in the live tool set makes "the lab cannot" a checked claim, not an assumption.

## ADR-032: Phantom-work guard — a capability-build proposal checks whether the capability already exists

- Status: accepted
- Date: 2026-06-14
- Incident (production): the false gap of ADR-031's decision#910 spawned branch#911 — a proposal to BUILD a capability (`get_file`) that already existed. The chain ran: misroute → false absence → a branch proposing to build what the lab already had. Fixing the verdict (ADR-031) addresses the assertion, but the proposal it spawned is a separate defect: phantom work the operator must still notice and decline.
- Decision: a capability-BUILD proposal (a branch whose intent proposes to build/extend a pack for capability X) must check whether X already exists before being proposed. At both branch-proposal sites (`plan` from a site claim, `interpret`'s follow-up child), `behaviors._phantom_capability` fires when a build verb, a capability noun, AND an alias for an AVAILABLE tool (`_EXISTING_CAPABILITY_ALIASES`, keyed to the live `_available_tools` set) all appear; if so the branch is NOT created — instead a `phantom_work_suppressed` observation is recorded naming the existing tool and flagging the prior gap as spurious. A build-proposal for a genuinely absent capability is untouched and still proposes.
- Fixture: `phantom_work` — a follow-up proposing to build `get_file` (live) is suppressed with a `phantom_work_suppressed` observation linked to the parent branch (no child branch); a follow-up proposing a genuinely absent sandbox capability still proposes the child branch.
- Rationale: an honesty invariant paired with ADR-031 — the self-check stops the lab asserting a false absence; the guard stops it ACTING on one. Together they cut the misroute → false-absence → phantom-build chain at every link. The guard is deliberately narrow (build verb + capability noun + a known alias for a live tool) so ordinary follow-ups and genuine build proposals pass; the alias map is documented and grows with the tool set.
- Maintenance guard (2026-06-14, Phase 6): "the alias map grows with the tool set" was a convention with nothing enforcing it — a tool joining `RESEARCH_WORKER_TOOLS` with no alias would let a phantom-build proposal for it through unnoticed (silent rot). The `alias_map_guard` fixture now asserts every `RESEARCH_WORKER_TOOLS` member is either aliased in `_EXISTING_CAPABILITY_ALIASES` OR named in a new documented exemption set `_ALIAS_EXEMPT_TOOLS` (and forbids entries that are both, or stale entries naming non-live tools), so adding a tool without classifying it fails CI loudly. The github list-* reads (`list_commits`/`list_issues`/`list_pulls`) are EXEMPT, not aliased: aliasing them would widen what the guard suppresses (a runtime behavior change), and no phantom build of a list-* read has been observed — the exemption is the explicit, test-enforced record of that choice. No runtime change; the guard's matching is untouched.

## ADR-033: Overclaim lint in drafts — the narrative is checked against the graph it cites

- Status: accepted
- Date: 2026-06-14
- Gap (external review): published posts used "five independent sources" over five SAME-AUTHOR artifacts and "fully autonomous" for one automated trigger inside an operator-steered pipeline. The project's founding sin is the narrative layer claiming more than the graph supports; the coverage check (ADR-029) catches UNFOOTNOTED claims, but a claim can be footnoted and still overstate what its own evidence shows. "Independent", "autonomous", "verified", "first ever" are the words that round the graph up.
- Decision: `draft_writer` gains an overclaim lint — `behaviors._overclaim_review`, a sibling to `_coverage_review`, run on the original body and appended to the review note. It flags overclaiming language whose claim the GRAPH EVIDENCE CONTRADICTS, for operator attention only — NEVER auto-blocked (the operator judges, the ADR-014 escape-hatch spirit; flagged phrases go in the review note). Each check is graph-grounded so a correctly hedged draft passes clean: (a) "independent"/"corroborated"/"separate sources" where the ≥2 cited evidence items all resolve to ONE origin signature (a source's author or URL site, an observation's joined source origins, else the creating actor) — same-origin artifacts are not independent corroboration; (b) "autonomous"/"unprompted"/"unsupervised" where an operator message or activation is in the branch's causal chain (the branch's `activation_message`, the `_OPERATOR_DIRECTIONS` registry, or a cited finding created by an operator) — operator-steered work is not unprompted; (c) "verified"/"proven"/"demonstrated" where NO cited evidence is a demonstration (an evaluation, or an observation whose category is measurement/benchmark/test/experiment/result) — descriptive observations do not earn proof language; (d) superlatives ("first ever", "the only/first/best/fastest", "unprecedented", "never before") in a paragraph carrying no evidence footnote. The grounding is the discriminator: a flag fires only when the graph actually undercuts the phrasing.
- Fixture: `draft_writer` Phase 1 — "independent" over same-author sources flags and over distinct-origin sources passes; "autonomous" over an `activation_message` branch flags; "proven" over a descriptive observation flags and over an evaluation passes; an unfootnoted superlative flags; a hedged-correctly draft draws no note.
- Rationale: the honesty family (ADR-026/029/031/032) — the lab must not slander its own capabilities, and it must not flatter its own results. ADR-029 made the footer record the real per-behavior model; this makes the body's claims answerable to the very evidence they footnote. The lint is advisory, not a gate: the operator still owns the publish decision, but now the inbox carries the overclaim alongside the draft instead of an external reviewer catching it after publication.

## ADR-034: MCP send_chat is commit-and-return — the append decides the request, the reply is read via get_branch

- Status: accepted; amends ADR-016 (its reply-wait note) and ADR-023 (the MCP arm of the chat-path failure domain)
- Date: 2026-06-14
- Gap (recurring production evidence): evt_14234, evt_16799, and others — the bounded reply-wait (`setting.mcp_reply_wait_seconds`, ADR-016 note) still lost under load. The comm_message append COMMITTED, but the operator's `send_chat` call timed out waiting for the answer behavior to finish on a busy worker, so the client saw a transport timeout for a request that had in fact succeeded. The bound was tuned under claude.ai's tool timeout, but "wait for the reply, just less" is still a wait that loses the race when the worker is saturated.
- Decision: `send_chat` is COMMIT-AND-RETURN-IMMEDIATELY. The comm_message append remains the ONLY step whose failure may fail the request (ADR-023, unchanged); once it commits, the tool returns at once with `status=accepted`, the committed `message_event_ids`, `message_id`, `thread_id`, and a poll hint. There is NO bounded wait and NEVER a timeout error after a successful append — the reply (the answer behavior, or a steering verb's confirmation) is fired onto the runtime worker fire-and-forget (`_submit_to_worker`, the ADR-023 decoupling: client fate is irrelevant to the reply's completion) and is READ via `get_branch`. This supersedes the bounded-wait approach entirely: `status=ok`-with-reply and `status=reply_pending`-on-timeout are both gone from the MCP path; `accepted` is the single success status (a post-commit upkeep degradation still returns `accepted`, now carrying the sanitized `degraded` steps). The HTTP `POST /chat` path is unchanged — it has no client tool-timeout constraint and still composes its reply inline.
- The retired setting: `setting.mcp_reply_wait_seconds` is no longer read by any code path. To avoid editing the kernel manifest (`lab_pack/kernel.py`'s seam-eligible whitelist is kernel — ADR-012), the field and its whitelist entry are KEPT IN PLACE as a documented no-op rather than removed; nothing consults the value, and the LabSettings field's description says so. Removing it outright is a future kernel change, not this one.
- Fixture: `scripts/test_mcp.py` — send_chat returns `accepted` with the committed event ids regardless of worker load (the reply job is captured fire-and-forget, not run inline), no reply lands while the worker is busy, and once the worker catches up the reply lands EXACTLY ONCE and is readable via `get_branch`. The archived-branch refusal/resurrection and the round-trip now read the reply via `get_branch` (the advertised poll path), not the send_chat response. `scripts/test_chat_robustness.py` re-asserts the append-failure-is-structured and post-commit-degrades-to-`accepted` arms.
- Rationale: ADR-023 already established that the append is the commit boundary and everything after it is decoupled from the client; the bounded wait was the one place the MCP path still pretended the reply was part of the request. Removing the wait makes the success criterion honest — "your message is committed" is exactly what `accepted` claims, and the reply lives where every other piece of lab state lives: in the log, read through a projection. A request can no longer time out after succeeding.

## ADR-035: GitHub rung 2 — the code_worker, the repo sandbox, and the submit_pr decision (the self-repair loop)

- Status: accepted; builds rung 2 of ADR-022 (designed there, not built until now); the queue's item 5 in ROADMAP.md
- Date: 2026-06-14
- Numbering note: ADR-022 rung 2 was sketched as a future "submit_pr"; the build session's brief called this ADR-033, but ADR-033 (overclaim lint) and ADR-034 (send_chat commit-and-return) had already landed, so this is ADR-035. The content is the rung-2 design, made real.
- Context: the lab repeatedly hits bugs that need a repo-level fix only the operator could make via Claude Code. This ADR closes that loop: the lab drafts a fix, PROVES it in a sandbox, and opens a pull request the operator approves on GitHub — two human gates, never auto-merge.

- Decision (Phase 1), the repo sandbox (`lab_pack/repo_sandbox.py`, PLUMBING): a bounded execution sandbox for cloning and running ALLOWLISTED repos. Built on the graph-code sandbox's isolation pattern (subprocess, wall-clock timeout, clean env — ADR-012 Phase 5), hardened and generalized:
  - CLEAN ENVIRONMENT — the sandbox subprocess inherits NO secret. Its env is built from scratch from a curated allowlist (`_ENV_ALLOWLIST`: PATH, HOME, locale, CA bundles — what git/pip need), never a wholesale copy of `os.environ`; `clean_env()` refuses to launch if any known secret key (ANTHROPIC_API_KEY, LAB_OPERATOR_TOKEN, LAB_MCP_TOKEN, GITHUB_TOKEN, GITHUB_WRITE_TOKEN, DATABASE_URL / LAB_DATABASE_URL, OPENAI_API_KEY) leaked into the allowlist.
  - WALL-CLOCK + RESOURCE LIMITS — every run is killed at `setting.sandbox_timeout_seconds`; on POSIX, CPU-time / address-space / core rlimits are set in a preexec hook (best-effort; the wall-clock bound is the primary kill).
  - ALLOWLISTED REPOS ONLY — checked against GITHUB_REPO_ALLOWLIST (the same allowlist github_read enforces) BEFORE any I/O.
  - CAPTURED OUTPUT IS EVIDENCE — stdout/stderr (truncated), exit code, duration, timed-out flag come back structured.
  - SUBPROCESS-HARDENED TO START; E2B is the documented future upgrade for running UNTRUSTED code (a hosted micro-VM: separate kernel + network namespace per run). This session takes NO E2B dependency — for now the sandbox runs only the lab's OWN allowlisted repos, where a clean-env subprocess plus the allowlist is the right floor. A subprocess shares the host network namespace, so this floor cannot ENFORCE "network only for clone + package install" — that enforcement is exactly what E2B buys; the allowlist (own repos only) + clean env (no credential to exfiltrate) are the mitigations until then. `run_repo_task` is the seam an E2B backend would slot behind.
  - HARD GATE (`scripts/test_sandbox_isolation.py`): sentinel secrets planted in the PARENT env; the sandbox runs a command that dumps its env and tries to read each secret; the gate asserts NONE is reachable inside and NONE appears in any captured output (the clone is stubbed so the RUN step exercises the real subprocess env). This test gates the phase — a sandbox that leaks credentials stops the build.

- Decision (Phase 2), the code_worker (`lab_pack/code_worker.py`, PLUMBING — mirrors the research worker, ADR-020): reacts to core tasks routed `codebase.code_task` (the lane that had NO reactor — every code_task recorded a capability gap). Two reactive stages over one log: `code_intake` (plain behavior) claims a routed task, resolves a code-task spec (repo, command, optional ref, optional fix diff — from `task.metadata.code_task` or parsed from the intent/operator direction), runs it in the sandbox (clone an allowlisted repo, run the command; for a fix-task apply the diff and re-run to PROVE it), and writes the captured output as ATTRIBUTED evidence (a `code_run` observation linked to a repo source) plus a code-synthesis request. `code_worker` (llm_behavior, model `setting.model.code_worker`, default `claude-opus-4-8` — reasoning over a change and its test output is top-tier per the operator's tiering) summarizes the run honestly and COMPLETES the task — done if the deciding run exited 0 (proven in the sandbox), failed otherwise; the verdict is the RUN'S, computed by the lab, never the model's. Run cap is `setting.code_run_cap`; the per-run wall-clock is `setting.sandbox_timeout_seconds`; the LLM stage rides the LabProviderWrapper (pause + daily cost cap) like every lab llm_behavior. Failure carries the error IN the event. Enablement: `code_worker_enabled` defaults False (no fixture clones by surprise); the server boot enables it; droppable like the research worker. An explicit `code_task` spec on a branch is itself a routing signal — it dispatches `codebase.code_task` regardless of the intent's verb (a "run the tests" task carries no code-WRITE verb but is still the code worker's lane).
  - branch#847 safety net (ADR-031 addendum): a read-to-verify task that lands in the codebase lane (the production regression shape — the deployed router had reverted to the keyword version; origin/main's verb classifier correctly routes branch#847's exact intent to research) is answerable EITHER way. Fixed routing sends it to research.deep_research (which carries get_file); OR, if it still reaches the code lane, `code_intake` sees there is no command to run, recognizes the read-to-verify intent (the corrected router), records a `routing_miss` naming research.deep_research, and fails with an actionable verdict (interpret fires) instead of dying in a dead lane.

- Decision (Phase 3), the submit_pr decision kind (the gate, the whole point): a code change the lab wants to LAND becomes an artifact (kind=`code_change`: the unified diff for the human reviewer + the concrete changed-file contents to commit) plus the sandbox PROOF (the code_worker's `sandbox_proven` evaluation, cited in `evidence_refs`). The lab opens a decision kind=`submit_pr`, status=pending. ONLY on operator approval (the FIRST gate) is a GitHub WRITE token exercised to push a branch and open a PR; the operator then reviews and merges on GitHub (the SECOND gate, a different system). No auto-merge, ever.
  - The write token: `GITHUB_WRITE_TOKEN`, a SEPARATE secret from the read-only `GITHUB_TOKEN` (read rate-limit access must not carry write scope), fine-grained and scoped to the lab repo only, ABSENT from the deployment until the operator configures it — absent means the lab can draft + sandbox + propose, and the operator opens the PR manually from the diff. Its presence is never required. The token is read ONLY inside `lab_pack/github_write.py`, rides ONLY in a request Authorization header, and is NEVER in the graph, any log line, or any payload — sentinel-audited (`scripts/test_public_safety.py` plants it and exercises an approved submit_pr through the gate; `scripts/test_sandbox_isolation.py` plants it among the sandbox sentinels). `_scrub` redacts it from any error text as defense in depth.
  - submit_pr is EXCLUDED from MCP exactly like approve/reject (the inbox is human-only) — annotate is fine, opening the PR is the operator's tap. (Verbatim per the build brief.) No MCP tool resolves a decision or opens a PR; `annotate_decision` may attach public commentary to a pending submit_pr without resolving it. The human operator resolves over `POST /lab/decision` (the inbox) or chat approve (non-MCP); MCP-tagged approve/reject stay refused (ADR-025).
  - Rejection records the operator's rationale (a `pr_rejected` observation); NO write token is touched. Approval that cannot open the PR (no token, clone/apply/API failure) records a `pr_failed` observation and the operator opens it manually — the diff is in the artifact.
  - Fixture (`submit_pr` in run_fixtures + the MCP block in test_mcp): a submit_pr pending → approve → (mocked) PR opened with the branch/diff, the write token's one exercise point joins the sentinel corpus → a second submit_pr → reject → no token exercised; the whole graph is asserted free of the write-token sentinel; the MCP tool surface is asserted to carry no decision-resolution/PR-opening tool while annotate IS available.

- Decision (Phase 4), the self-repair loop, wired and documented (ROADMAP/ARCHITECTURE): a production failure → finding observation → the lab proposes a fix (a code_task in the codebase lane) → the sandbox proves it → a submit_pr decision → operator approves → PR → operator merges → Replit pulls. This session builds and PROVES the rails (every phase fixtured, both sentinel gates green); it does NOT auto-open a PR — the first real exercise is operator-chosen later, and `propose_submit_pr_fn` is the rail the operator (or a future auto-propose step) calls. This is the rung that turns "us touching GitHub" into "approving PRs the lab opened."

- Rationale: the lab's mission needs repo-level fixes to be in the loop, and the loop's missing miles were exactly the dangerous ones — running code, and writing to GitHub. The sandbox makes running code bounded and credential-free (the Phase-1 sentinel gate); the submit_pr decision makes writing to GitHub doubly-gated and token-clean (the Phase-3 sentinel gate). The same posture as graph code behind `LAB_ALLOW_GRAPH_CODE` and the read path behind the allowlist: the write path is specified, fixtured, and dark until the operator lights it.

## ADR-036: Self-dispatched code repair — the planner turns the lab's OWN observed defects into gated code-fix proposals (closing the self-repair loop)

- Status: accepted; builds on ADR-035 (the code worker, the repo sandbox, the submit_pr decision) and ADR-031 (the capability self-check's routing_miss verdict)
- Date: 2026-06-14
- Context: ADR-035 built and proved the self-repair RAILS — clone, run, prove a fix, open a doubly-gated `submit_pr` — but left the loop's FIRST mile to the operator: `propose_submit_pr_fn` was "the rail the operator (or a future auto-propose step) calls." The lab recorded its own code defects (a `routing_miss` — the capability self-check's OWN verdict that a task was a routing/code bug, not a capability absence, ADR-031; a finding the build diagnosed) but did NOTHING with them. The goal is lab-repairs-lab, not operator-hands-lab-each-bug. This ADR closes the first mile: the PLANNER self-dispatches a code-fix proposal from the lab's own observed failures, and the code worker carries it to a pending submit_pr. It is a real authority expansion — the lab now decides, unaided, that one of its own defects is worth a repair branch — so it is recorded here with its bounds.

- Decision (Phase 1), self-dispatch: a new behavior `self_repair` (`lab_pack/behaviors.py`) reacts to the lab's own defect observations and proposes a GATED code-fix branch routed `codebase.code_task`, carrying the defect's evidence (the observation/evaluation ids) as the task's intent. It fires for two concrete, self-classified defect shapes about the lab's OWN code:
  - a `routing_miss` observation — concrete by construction (it names the misrouted lane and the task; ADR-031 already classified it a routing/code bug, not an absence);
  - a finding the build EXPLICITLY tagged `metadata.code_defect` (an OPEN defect about the lab's own code, with `repo` + a `fix_command` proving harness, optionally a candidate `fix_diff`). Free-text cue-matching is deliberately NOT used: most findings DESCRIBE already-fixed bugs ("Fixed by …"), so cue-matching would re-propose phantom repairs (the ADR-032 hazard). The branch carries a `code_task` spec ({repo, command, propose_pr:True, diff?}) — itself the routing signal (ADR-035) — and is `status=proposed`/`authority=gated`: the operator activates it through the existing dispatch path, exactly like any branch. The code worker then clones the allowlisted repo, proves the fix in the sandbox, and (Phase 2 below) opens a gated submit_pr. observation#1046 — the source-selection bug the lab logged itself (the research worker's `_source_urls` orders operator-named URLs behind the derived defaults, starving them under a tight fetch cap — the inverse of its documented intent) — ships as a `code_defect`-tagged LIVE_FINDING and is the worked first case: after deploy the planner proposes its fix unaided.

- Decision (Phase 2), the operator front door (secondary, the convenience): a `fix` steering verb (`_apply_steering`) so the operator CAN also hand a bug in directly — it proposes a code-fix branch from the message and activates it, the SAME downstream path as Phase 1's self-dispatch. `code` is deliberately NOT an alias: it is overwhelmingly the noun ("verify the code"), the exact collision ADR-025's routing avoids. This is assisted repair, explicitly secondary to Phase 1. The code worker's PR last mile: a PROVEN fix the lab was asked to LAND (the `propose_pr` flag, set by both paths) becomes a PENDING `submit_pr` carrying the diff + the post-fix file states the sandbox produced (`repo_sandbox.run_repo_task` now reads back a proven diff's target files), citing the run's `sandbox_proven` evaluation — the loop's last mile, formerly operator-only.

- Decision (Phase 3), the guardrails (so self-dispatch does not run wild):
  - OWN repo only — a self-proposed fix targets ONLY `yoheinakajima/activegraph-lab` (the lab's own repo). An upstream-tagged finding ("(upstream …") or a `code_defect` finding naming any other repo (even an allowlisted one like activegraph-packs) is NOT self-dispatched (CONTRACT.md / ADR-005: the lab never edits the packs repo or core). Enforced at the proposal (`_self_repair_target` / `is_own_repo`) AND independently at the code worker's PR mouth (`_maybe_propose_pr`) — defense in depth.
  - submit_pr stays human-gated + MCP-excluded — the code worker only opens a PENDING decision; opening the PR is still the operator's tap, and approve/reject/submit_pr remain refused over MCP (ADR-035 unchanged). Every PR is your tap.
  - a failed proof does NOT submit_pr — only a sandbox-PROVEN fix opens a submit_pr; a fix that fails its own sandbox proof records the failure (the task is rejected) and opens nothing. A self-proposed fix must earn its PR.
  - a CAP on concurrent self-repair branches — `setting.max_self_repair_branches` (default 3, NOT seam-eligible: a guardrail on self-modification stays in git, not in a gated decision the lab could raise on itself; 0 disables self-dispatch). Over the cap, the planner idles and records one `self_repair_capped` observation; the defect stays in the log to revisit.
  - LIVE LANE only — self-repair is dark unless the code worker is enabled (proposing a fix into a dead lane would be the very dishonesty ADR-031 forbids).

- Fixture: `self_repair` — a routing_miss about the lab's repo → the planner proposes a gated codebase.code_task branch; observation#1046 → the planner proposes its fix UNAIDED (the canonical case); the full loop (a tagged defect with a proven fix → clone → prove → a PENDING submit_pr citing the sandbox proof, write-token sentinel-clean, no PR opened); the operator `fix` verb path; and the guardrails (a non-own-repo defect is NOT self-dispatched; a fix that fails its proof opens no submit_pr; the concurrency cap holds). The pre-existing code_worker / submit_pr / sandbox-isolation sentinel gates stay green.

- Rationale: this is the rung that turns the self-repair loop from "rails the operator hand-cranks" into "the lab proposes its own repairs and the operator approves the PRs." It is a genuine authority expansion — self-dispatch is the lab acting on its own diagnosis — so it is bounded the same way every prior expansion was: narrow triggers (only the lab's own self-classified defects), the own-repo allowlist, the human-gated + MCP-excluded PR, a proof requirement, and a concurrency cap. The honest defaults hold: dark unless the lane is live, and every PR is still the operator's tap.

- Amendment (branch#935 — structured `code_task` into the worker): production showed a self-repair branch the planner proposed activate, dispatch to `codebase.code_task`, REACH the code worker (routing works), then fail — `code_worker: no command could be resolved for this task`. The proposal emitted a prose intent but the code worker had nothing RUNNABLE. The fix is two-sided. THE REAL FIX (the proposer): `_propose_code_fix_branch` writes a complete `code_task` spec — `{repo, command, propose_pr, brief}` (the candidate `diff` too when present) — that `_dispatch_branch` copies VERBATIM onto `task.metadata.code_task`, with `repo` defaulting to `LAB_OWN_REPO` and `command` to the lab's own proof harness (`python -m lab_pack.fixtures.run_fixtures`, `_SELF_REPAIR_DEFAULT_COMMAND`); the prose bug description rides as the authoring brief on both `task.metadata.code_task.brief` and `task.metadata.brief`. DEFENSE IN DEPTH (the worker): `code_intake` now DEGRADES GRACEFULLY — a task that arrives with a repo resolved (from metadata OR the intent) but no command, where the repo is the lab's OWN allowlisted repo and the intent is genuine code-write work, infers the default proof command and runs rather than hard-failing (`command_inferred` is stamped on the claim observation). The branch#847 read-to-verify net is untouched (a research-routed intent still records a `routing_miss` to `research.deep_research`, never an inferred command), and a GENUINELY malformed task — no repo, no command, nothing to infer from — still fails honestly with the same "no command could be resolved" verdict. Fixtured: `self_repair` asserts the dispatched task carries `code_task={repo,command}` + a brief (both the planner and the `fix`-verb paths); `code_worker` adds the branch#935 net (bare own-repo task → default command inferred → canned clone+command runs, no "no command resolved") and the honest-failure case. No invariant changes — this hardens ADR-036's implementation so the loop the operator activates actually reaches a running code worker.

## ADR-037: Diff authoring — the code_worker writes the fix, not just runs it (the self-repair loop's last mile)

- Status: accepted; completes ADR-035 (the code worker / repo sandbox / submit_pr) and ADR-036 (self-dispatch + the `fix` verb)
- Date: 2026-06-14
- Context: ADR-035/036 built the self-repair loop UP TO authoring. Production proved it end to end except the last mile: the `fix` verb created branch#1630 with `code_task={repo,command,brief}`; the code worker cloned `yoheinakajima/activegraph-lab` and ran the real suite (35/35, 11.96s); but it only RAN the existing suite — it did NOT author a diff for the source-ordering defect or add a regression test (decision#1648, observation#1647). The capability gap was exact: clone + run + PROVE worked; WRITE-the-fix did not. A propose_pr task with a brief but no candidate diff just re-ran the proof command (and, with no diff to land, opened no PR — the honest dead end). This ADR closes it: the code worker now AUTHORS the diff.

- Decision (Phase 1), the authoring stage (`code_author`, a new llm_behavior in `lab_pack/code_worker.py`): a propose_pr task that carries a BRIEF but NO candidate diff is an AUTHORING task (decided in `code_intake`, surfaced as `metadata.authoring` on the claim observation). `code_intake` reads the relevant files from the clone (`repo_sandbox.clone_and_read` — a clean-env `git clone` + a bounded file read, no command run, so no run budget spent and no command env exercised) and emits a `code_authoring_request`. `code_author` (model `setting.model.code_worker` — authoring a fix is top-tier reasoning, the same plane as synthesis; routed via the `code_author → code_worker` model alias) authors a unified diff from the brief + the relevant files, the lab applies it in the sandbox and runs the proof command, and the RUN decides success — never the model, exactly as ADR-035 holds for synthesis.

- Decision (Phase 2), the bounded retry + the two outcomes:
  - tests PASS → emit the existing `code_synthesis_request` carrying the AUTHORED diff + the proven run (with the post-fix file states the sandbox read back), so the unchanged `code_worker` synthesis stage opens the gated, human-gated, MCP-excluded `submit_pr` (ADR-035/036's last mile, reused not duplicated).
  - tests FAIL → feed the captured failure output back and retry authoring, up to `setting.code_author_max_attempts` (default 2) total attempts; still failing → record an honest `authoring_unproven` evaluation ("authored a diff but could not make it pass after N attempts"), reject the task (so interpret fires), and open NO `submit_pr`. A fix must EARN its PR by proving in the sandbox.
  - The authoring step must add or extend a regression test when the brief describes a defect (the prompt requires it; the brief already asks for one).

- Decision (Phase 3), the guardrails — UNCHANGED from ADR-035/036, and now also covering the authored path: own allowlisted repo only (`_maybe_propose_pr` still independently checks `is_own_repo` at the PR mouth); `submit_pr` human-gated + MCP-excluded; the sandbox secret-isolation sentinel still passes with the authoring step present (`clone_and_read` adds only a clean-env clone + file read — no new env reaches a subprocess); inert authoring output (LLM budget/pause/parse) records an honest gap and opens no PR; a failed authoring run opens no PR. The LLM authoring rides the `LabProviderWrapper` like every lab llm_behavior (pause + the daily cost cap gate it there).

- Fixture (`code_authoring` in run_fixtures, LLM authoring MOCKED — the mock reads the failing check in the file context and authors the flip + a regression test): brief → authored diff (fix + regression test) → applied → proof passes → a PENDING `submit_pr` carrying the AUTHORED diff + the proven post-fix file contents + the cited `sandbox_proven` evaluation, no PR opened; brief → an unprovable proof command → the authored diff fails → bounded retry (2 attempts) → an honest `authoring_unproven` gap, the task rejected, NO `submit_pr`. ## ADR-038: Authoring emits full file content, the lab builds the patch — stop asking the model for a unified diff

- Status: accepted; amends ADR-037 (the `code_author` contract). The loop's shape, bounds, gates, and the rest of ADR-037 are UNCHANGED — this changes only WHAT the model emits and WHO computes the patch.
- Date: 2026-06-15
- Context: ADR-037 had `code_author` emit a hand-written unified diff (`--- a/… / +++ b/… / @@ … @@`), which the lab applied with `git apply`. Production proved this fails exactly where the literature says it does: branch#1667 — the code worker authored a CONCEPTUALLY CORRECT fix (reordered operator URLs ahead of defaults, added the regression test, evt_35262) but BOTH attempts failed `git apply exited 128: corrupt patch at line 16` / `No valid patches in input` (observation#1675/#1680). The model computed the `@@` hunk headers (offsets and line counts) by hand and got them wrong — the standard LLM diff-authoring failure. The intent was right; the patch MECHANICS were wrong. Asking a language model to be a patch calculator is the bug.
- Decision: the `AuthoredDiff` contract changes — `code_author` emits, per changed file, the FULL new file content (`AuthoredDiff.files: [{path, content}]`), NOT a unified diff. The lab constructs the patch DETERMINISTICALLY: `run_repo_task(new_files=…)` reads each cloned original, diffs it against the model's new content with `difflib.unified_diff` (lines normalized to end in a newline, `--- /dev/null` for a new file), and applies the result with `git apply`. The patch is git-apply-valid by construction — there is no hand-computed hunk header left to be wrong. The git-built patch (`result["diff"]`) is what rides to `submit_pr`; the model supplies INTENT (the file as it should read), the tooling supplies MECHANICS (correct offsets, counts, headers).
- Unchanged from ADR-037: the RUN still decides success (apply + re-run the proof command — `proven` iff the deciding run exits 0); the bounded retry feeds the captured apply/test failure back (`setting.code_author_max_attempts`, default 2); a defect brief still requires a regression test (now emitted as that test file's full content); honest-gap-on-failure (`authoring_unproven`, the task rejected, NO `submit_pr` — a fix earns its PR by proving); every guardrail (own allowlisted repo only, doubly-gated + MCP-excluded `submit_pr`, the sandbox secret-isolation sentinel, the clean-env clone). The supplied-candidate-diff path (ADR-036, a task that arrives WITH a `diff`) is untouched — `new_files` wins only when both are present.
- Fixture (`code_authoring`, LLM authoring still MOCKED — the mock now emits full file CONTENT, never a diff): the authoring leg additionally asserts the CONSTRUCTED patch applies with `git apply --check` (the branch#1667 failure is gone by construction). The pass/fail legs, the `submit_pr` carrying the git-built diff + the proven post-fix file contents, the bounded-retry honest gap, and the sandbox-isolation / public-safety / MCP sentinel gates stay green (36/36 fixtures).
- Rationale: removing hand-computed hunk headers removes the single mechanical step the model was reliably getting wrong, without weakening any bound — the model still authors the fix (intent), the sandbox still proves it (verdict), and the PR is still the operator's tap. Full file content is the smallest contract that lets the tooling own patch mechanics; labeled search/replace blocks were the alternative but add a parser and a new failure mode (a block that doesn't match) for no gain here.

## ADR-039: Self-repair proof integrity — execute the authored test, author over the full file, classify the PR failure

- Status: accepted; hardens ADR-035/036/037/038 (the self-repair loop). No invariant is reversed — the loop's shape, bounds, gates, and the human-gated + MCP-excluded `submit_pr` are UNCHANGED. This closes three integrity gaps the loop's FIRST real PR attempt (branch#1704) exposed, in the order they compounded.
- Date: 2026-06-15
- Context: branch#1704 ran the loop end to end and hit three walls in sequence, each a quieter dishonesty than the last:
  1. TRUNCATION — the authoring step's view truncated the target file (`research_worker.py`); the worker noted "the source was truncated in the request," could not author the real `_source_urls` fix, and fell back to a TEST-ONLY change (a new `lab_pack/tests/test_source_url_ordering.py`, modifying no source).
  2. FALSE GREEN — the proof command (`python -m lab_pack.fixtures.run_fixtures`) reported 36/36 pass for that no-op diff, because the fixture runner does NOT collect pytest files under `lab_pack/tests/`. The authored regression test was NEVER EXECUTED, yet the proof was green — so a no-op fix opened a `submit_pr`.
  3. CREDENTIAL WALL — the approved `submit_pr` then failed to open a PR: "could not read base ref 'main': Bad credentials" (a bad `GITHUB_WRITE_TOKEN`), recorded as a generic `pr_failed` indistinguishable from a code/git error.
  The chain: truncation → test-only fix → unexecuted test → false green → `submit_pr` for a no-op → credential wall. Each phase below cuts one link.

- Decision (Phase 1), PROOF INTEGRITY — execute the authored test: when an authored change adds or modifies a test, the proof MUST actually run that test, and the fix is `proven` ONLY if the test is EXECUTED AND PASSES. `code_author` collects the authored test paths (`_is_test_path`: a `.py` named `test_*`/`*_test`, or any `.py` under a `tests/`/`test/` dir; `conftest.py` excluded) and passes them to `run_repo_task(test_paths=…)`, which — after the proof command passes — runs them DIRECTLY with `python -m pytest <paths> -p no:cacheprovider -q` (the `-m` form puts the clone root on `sys.path`). `_regression_executed_and_passed` requires pytest exit 0 (it exits 5 on "no tests collected" — the unexecuted-test case) and guards the "no tests ran" string. A test-only no-op fix now fails its OWN regression test against the unfixed clone → not proven → an honest `authoring_unproven` gap → NO `submit_pr`. A self-repair whose regression test never ran can no longer confabulate a green proof. `pytest>=7.0` joins the dependencies so it is present in the runtime the sandbox clones into.
- Decision (Phase 2), FULL-FILE AUTHORING CONTEXT: the authoring step receives the FULL target file, never a truncated brief. `clone_and_read` reads the hint (target) files in FULL — no per-file cap, no `_truncate`, always included — and bounds only the EXTRA context files by the budget; `_authoring_request_text` renders each relevant file uncut (the `[:6000]` slice that starved branch#1704 is gone). The model now sees the entire file it must reproduce, so "the source was truncated" cannot recur.
- Decision (Phase 3), PR-FAILURE VISIBILITY: a `pr_failed` observation now carries a `failure_class` (`_classify_pr_failure`) distinguishing a CREDENTIAL wall ("Bad credentials" / 401 / 403 / unauthorized / forbidden / "resource not accessible") from a `rate_limit` 403 and from any `other` git/API error, and the recorded text states the class plus the operator's next action (replace `GITHUB_WRITE_TOKEN` and re-approve, vs open manually from the diff). The honest `pr_failed` is kept (no retry-from-MCP — the next fix-dispatch is the test); the classification makes the next attempt's outcome self-classifying: it clears (token fixed) or recurs as `credentials` (still bad).
- Unchanged: the RUN still decides success (never the model); the bounded authoring retry (`setting.code_author_max_attempts`); honest-gap-on-failure with NO `submit_pr`; every guardrail (own allowlisted repo only, doubly-gated + MCP-excluded `submit_pr`, the sandbox secret-isolation sentinel, the clean-env clone); the supplied-candidate-diff path (ADR-036). The write token still rides only in a request header, never in any payload.
- Fixture (`code_authoring` + `submit_pr`): an authored change adding a regression test runs THAT test in the proof and FAILS if it would fail against the unfixed source (a test-only no-op → unproven → no PR); a fix modifying source AND adding a passing test proves green ONLY when both the proof command and the new test pass; the authoring context contains the FULL target file (a >6000-char target appears uncut); a credential PR failure is recorded as a distinct classified risk (`failure_class=credentials`), an other-error as `failure_class=other`. The pre-existing self-repair / sandbox-isolation / public-safety / MCP sentinel gates stay green.
- Rationale: the loop was honest about mechanics (ADR-038 made the patch apply) but not yet about PROOF — a green check that proves nothing is worse than a red one. Executing the authored test makes "proven" mean what it says; the full-file context lets the model author the real fix instead of a test-only stand-in; the failure classification makes the operator's token replacement unambiguous. None of this expands authority — it makes the existing authority truthful.

## ADR-040: Red-then-green proof — a regression test must fail on the unfixed baseline before it can prove a fix

- Status: accepted; hardens ADR-039 (which hardened ADR-035/036/037/038 — the self-repair loop). No invariant is reversed: the loop's shape, bounds, gates, and the human-gated + MCP-excluded `submit_pr` are UNCHANGED. This closes a SECOND false-green hole the loop's production runs exposed, through a path ADR-039 Phase 1 did not cover.
- Date: 2026-06-16
- Context: ADR-039 Phase 1 made the proof EXECUTE an authored pytest test (a test that is never run cannot confabulate a green proof — branch#1704). But two production runs showed the deeper defect was not "unexecuted" — it was "proves nothing":
  1. branch#1704 (pytest): a test-only no-op. ADR-039 caught it because the authored test, run after the fix, FAILED (the source was never fixed).
  2. branch#1751 (FIXTURE): a second test-only no-op through a NEW hole. This run authored a fixture `.yaml` (`research_worker_source_order.yaml`) — NOT a pytest file. The proof command (`python -m lab_pack.fixtures.run_fixtures`) DOES run fixtures, so it executed and passed 36/36 — but passed VACUOUSLY against the UNFIXED code (`_source_urls` was never changed). The lab's own promote rationale (decision#1776) admitted it: "the actual fix to research_worker.py was not implemented — only the regression test was added." False-green moved from "unexecuted test" to "test that never fails against the bug."
  Root cause (covers BOTH branch#1704 and #1751): the proof could not tell a REAL regression test (red on the unfixed baseline, green after the fix) from a test that passes regardless. A test green on the baseline proves nothing — whether it is unexecuted, vacuous, or simply does not exercise the defect.
- Decision, RED-THEN-GREEN: when a self-repair authors or modifies a test — a pytest file (`is_pytest_test_path`) OR a fixture `.yaml` under a fixtures/ dir (`is_fixture_test_path`); together `is_authored_test_path` — the sandbox proves it in TWO phases. (1) RED: clone a PRISTINE copy, apply ONLY the authored test files (the test-only sub-diff, `_filter_diff_to_paths` over the per-file fragments of the built/supplied diff), and run the test; it MUST FAIL (`_regression_is_red`: ran AND reported a failure — pytest exit 5 / "no tests collected" is NOT a red, a pass is NOT a red). (2) GREEN: apply the FULL diff and run the test again — it MUST PASS (`_regression_executed_and_passed`) — and the full suite (the proof command) MUST be green. `proven = baseline_red AND afterfix_green AND suite_green`. A separate clone (not a git reset) runs the red phase, so it works against the canned fixture clones (which are not git repos) and against live clones alike.
- Decision, BOTH FORMS, ONE RULE: the fixture `.yaml` form is held to the SAME standard as pytest. A pytest test is exercised directly (`python -m pytest <paths>`); a fixture is exercised by the proof command itself (the suite runner is what runs a fixture). Either way the authored test, run against the unfixed baseline, must go red. A vacuous fixture that passes regardless (branch#1751) is now an honest `authoring_unproven` gap — "the authored regression test did not fail against the unfixed baseline — it proves nothing" — with NO `submit_pr`.
- Decision, DERIVE-FROM-DIFF: when the caller names no test paths but the diff itself adds a test (a supplied candidate diff, ADR-036), the authored-test files are derived from the diff's `+++` targets, so a supplied diff that ships a test is red-then-green too. A source-only fix (no authored test) keeps the existing rule: the full suite must be green. A DEFECT brief that authors no regression test therefore cannot reach "proven" through a baseline-green test — it has no test to go red.
- Unchanged: the RUN decides success, never the model; the bounded authoring retry (`setting.code_author_max_attempts`); honest-gap-on-failure with NO `submit_pr`; every guardrail (own allowlisted repo only, doubly-gated + MCP-excluded `submit_pr`, the sandbox secret-isolation sentinel, the clean-env clone). The red phase adds sandbox runs (a second clone + the test); `code_run_cap` bounds the per-task budget as before.
- Fixture (`red_green_proof`): a real source fix + a FIXTURE that is red-on-baseline and green-after → a PENDING `submit_pr`; a real source fix + a VACUOUS fixture (green on the baseline) → `authoring_unproven`, NO `submit_pr` (branch#1751 reproduced); a test-only no-op (pytest) red-on-baseline-and-after → `authoring_unproven`, NO `submit_pr` (branch#1704 reproduced); a source fix whose test STAYS RED after (wrong assertion) → unproven, NO `submit_pr`. The `code_authoring` legs are upgraded to author REAL red-then-green tests (a vacuous `assert True` is now correctly unproven); the pre-existing self-repair / sandbox-isolation / public-safety / MCP sentinel gates stay green.
- Rationale: ADR-039 made "proven" mean "the test ran." A test that runs but passes regardless is the same lie one layer down — and the fixture form let it back in. Red-then-green makes "proven" mean what a fix actually owes: a regression that the bug breaks and the fix repairs. The proof is now a control, not a formality — it can fail, and a self-repair that cannot make it fail-then-pass earns no PR. None of this expands authority; it makes the existing authority truthful.

## ADR-043: Parse-guard — an unparseable authored file can never become a patch or a PR (the branch#1859 truncation)

- Status: accepted; hardens ADR-042 (the `code_author` → `materialize_authored_files` → `build_diff_from_new_files` → proof path). Adds one unconditional gate; changes no authority, no schema, and no other step.
- Date: 2026-06-17
- Context: ADR-042's verification ran the `_source_urls` reorder end to end and reported a clean bounded-edit `submit_pr`. PRODUCTION did not match: the self-repair branch behind PR #32 committed a TRUNCATED `lab_pack/research_worker.py` — the full 588-line / ~26k-char file cut to 438 lines, ending mid-statement on a literal `…[truncated, 26366 chars total]` marker baked into the source. The file raises `SyntaxError` on import; merging it to `main` (merge 59ccf2a) broke the lab's boot. Two things had to go wrong together for that marker to reach `main`:
  - **The model emitted whole-file `content`, not bounded `edits`.** ADR-042 lets `AuthoredFile` carry EITHER `edits` (modify) OR `content` (create), and `materialize_authored_files` treats `content` on an EXISTING file as a "whole-file rewrite" (`new_files[rel] = content`, the `elif content is not None` branch). So when the model chose to rewrite `research_worker.py` as full `content` instead of a search/replace edit, the materializer replaced the real file with that text wholesale — search/replace never engaged for this file, and there is no bound or check on a whole-file `content`.
  - **The `content` the model echoed was already truncated.** `clone_and_read` reads HINT (named target) files in FULL (ADR-041), but any OTHER text source is CONTEXT and is capped: `f.read(max_total_chars)` then `_truncate(...)` at 20_000 chars, which appends exactly `…[truncated, N chars total]`. In this run `research_worker.py` was NOT resolved into the hint set (the brief named the FUNCTION `_source_urls`, not the file path), so the 26k target fell into the budget-bounded context fill and was shown to the authoring step truncated — marker and all. The model copied that truncated context blob back as its `content`. So the truncation entered through `clone_and_read`'s context cap, the materializer's unbounded `content` branch let it through, and the deterministic patch builder faithfully encoded "replace the whole file with the truncated text" — every downstream step did exactly what it was told. The proof command (`run_fixtures`) WOULD normally have caught it (a truncated module fails to import), but the run that produced PR #32 did not block on it, and a fix whose proof command never imports the broken module never would.
- Decision: a hard, unconditional PARSE-GUARD between materialization and patch-build. `repo_sandbox.parse_guard_error(new_files)` compiles (`compile(content, path, "exec")`) every authored `.py` and returns the first `SyntaxError` (file + message + line); `run_repo_task` calls it immediately after `new_files` is established (from `authored_files` materialization OR a supplied `new_files`) and BEFORE `build_diff_from_new_files`. A failure sets `result["error"]`, leaves `proven=False`, and returns — so no patch is built, no proof runs, and `code_author` falls into its existing failure path: the parse error rides `_failure_excerpt` into the bounded retry (the model is told its file does not parse and can revise), and on bounded-out it records an honest `authoring_unproven` gap and opens NO `submit_pr`. A truncated or otherwise unparseable file can no longer reach a PR, independent of WHICH upstream step (context truncation, a content-vs-edit choice, a bad materialization) produced it. The guard is the invariant; the upstream causes above are the diagnosis, not the gate.
- Scope held: only `.py` files are compiled (a non-Python authored file — a fixture `.yaml`, a Markdown doc — is not parse-checked here; its own proof leg covers it). The guard adds no authority, touches no schema, and leaves ADR-042's contract, the red-then-green proof (ADR-039/040), the one-predicate `submit_pr` gate (ADR-041), and every existing guardrail (own allowlisted repo only, doubly-gated + MCP-excluded `submit_pr`, the secret-isolation sentinel, the clean-env clone) UNCHANGED. The deeper upstream fix — keeping `clone_and_read` from showing a named target as truncated context, and bounding/forbidding a whole-file `content` over an existing file — is real and noted for a follow-up branch, but the parse-guard is the load-bearing backstop: even if the model is shown a truncated file again, the lab refuses to ship it.
- Fixture (`search_replace_authoring`, extended): unit — `parse_guard_error` rejects the exact branch#1859 truncated shape (the `…[truncated, N chars total]` marker), rejects a plain `SyntaxError`, passes well-formed Python, and ignores non-`.py` files; end-to-end — `run_repo_task` driven with unparseable authored `content` returns an error containing "does not parse", builds NO diff, and is NOT proven. The pre-existing legs and the full suite stay green (39/39 fixtures).
- Hotfix (separate from this guard, same incident): `main` was repaired by restoring `lab_pack/research_worker.py` from the pre-PR-#32 parent (1488b8f, the full parseable file) and re-applying ONLY the intended `_source_urls` reorder by hand — operator-supplied URLs (task description + `metadata.activation_message`) now lead the derived claim/mission defaults, after any `operator_direction`, then dedup-and-cap as before. The PR-#32 regression test (`lab_pack/tests/test_source_urls_order.py`) is kept and is RED against the pre-fix order and GREEN against the corrected file.
- Rationale: the self-repair loop's whole safety story is "a fix must PROVE before it earns a PR." A file that does not even parse cannot prove anything — yet one reached `main` because the proof command, not the patch, was the only thing standing between authoring and the operator's tap, and that command can be blind to a break it never imports. The parse-guard makes "it parses" a precondition of "it can become a patch," which is strictly weaker than proving and strictly cheaper than any import — the right altitude for a backstop that must never have a hole.

## ADR-044: The daily heartbeat — a bounded, gated, killable standing behavior (the first wall-clock cadence)

- Status: accepted
- Date: 2026-06-21
- THE DEPARTURE, stated plainly: every behavior before this one is REACTIVE — the log is the agent, there is no orchestrator and no scheduler (ADR-001..008; ARCHITECTURE.md "Twelve small reactive behaviors, no orchestrator"). The heartbeat is the FIRST external clock that pokes the agent on a timer. That is a deliberate architectural change, recorded here with its bounds. It does NOT make the lab orchestrated: there is still no coordinator and no queue; the heartbeat injects ONE unit of input per cadence window and then the existing reactive pipeline runs exactly as it always has. It is a standing INPUT source on a wall-clock cadence, not a control plane.
- Why a heartbeat at all: the lab's mission is to grow activegraph.ai's evidence base continuously, but a purely reactive lab only moves when an operator pokes it. The backlog (proposed branches, operator-noted directions, a site that changes) sits idle between operator visits. The heartbeat advances that EXISTING backlog by one step per day so the evidence base grows unattended — without inventing work and without touching the gate.
- The critical refinement (both external reviews flagged "cadence outruns evidence"): a bare planner tick on an idle graph just re-proposes from stale findings. So a heartbeat must bring in FRESH input BEFORE the planner reacts. Each tick rotates through a small operator-defined worklist (`heartbeat_worklist`) and does ONE step — bringing in new observations — then `run_until_idle` lets the reactive `plan`/`interpret`/`work` behaviors react to the NEW input. The heartbeat advances the backlog by one step; it never plans from nothing. Worklist vocabulary (cheap → expensive): `advance_branch` (activate the oldest proposed/scoped branch — the cheapest useful step, no crawl), `research_direction` (activate the oldest branch carrying an operator-noted direction — operator_direction/operator_note), `recrawl` (a fresh crawl of the mission site). Comma-separated; the heartbeat rotates from a cursor rebuilt from the last `heartbeat.fired` event's `worklist_index` and takes the first step that has work (so an empty step does not waste the window). Activation reuses the reversible operator-tier verb path (ADR-025): a `branch_activated` observation + a status patch the existing `work` behavior dispatches on. Nothing new dispatches.
- Wall-clock, not a loop (the idempotency contract): `lab_pack/heartbeat.py: maybe_heartbeat(rt)` is called from the SAME request/tick chokepoints as the stall watchdog (`check_stalls`) — the server's `/lab/feed` poll and a guarded call at boot. It computes the current cadence WINDOW (`heartbeat_cadence`: "daily" = one tick per UTC calendar day; "off" = disabled) and fires AT MOST ONCE per window. Every tick STAMPS its window key into the event payload, and the next tick is suppressed if any `heartbeat.fired` OR `heartbeat.skipped` event already carries the current window key. So the cadence is enforced off the DURABLE LOG, not process state — a mid-window restart does not double-fire (CONTRACT.md: no side database; the log is the persistence). It is not a self-retriggering loop: nothing the heartbeat emits triggers the next heartbeat; the next tick happens only when the next window opens and some unrelated activity calls the chokepoint.
- BUDGET, the spend floor: before ANY heartbeat work, spend is refreshed from the log (`sync_daily_budget`) and if `llm_cost_today` is at/above `heartbeat_budget_ceiling_usd` (default $15, well under the $50 daily cap and the $100 kernel ceiling, ADR-019), the tick records `heartbeat.skipped:budget` and does nothing. The heartbeat can never approach the daily cap on its own.
- THE GATE IS UNCHANGED — stated verbatim, as required: **everything the heartbeat causes still lands as PROPOSALS in the operator inbox. The heartbeat NEVER approves, promotes, or submits_pr. It cannot self-approve anything.** By construction it only (a) activates an existing branch and (b) queues a `crawl_request` the `ingest` behavior already reacts to — both feed the normal reactive pipeline whose every publish / self-modify / `submit_pr` last mile remains a human decision (the `gate` behavior, ADR-035). The heartbeat creates and resolves no decision, exercises no write token, and fires no approve/promote steering verb.
- Killable WITHOUT a deploy: set the `heartbeat_cadence` seam to `"off"` and the heartbeat stops instantly — it hot-loads through the gate like any seam, no deploy (`/lab/seams` shows the active value; an approved `setting.heartbeat_cadence` = "off" disables it the moment it loads). The global `pause_lab` (ADR-015, MCP/UI, no approval) also stops it — the heartbeat respects the pause like every worker. Two independent kills, neither needing a redeploy.
- AUDIT: every tick emits a `heartbeat.fired` or `heartbeat.skipped` marker event (the marker family, ADR-013/015/021 — projects no graph state) carrying the window, the cadence, the reason, the worklist + chosen index, and what it advanced (the step + target). The daily behavior is fully legible in the log.
- FAN-OUT WARNING (a real concern, and the follow-up): a daily `recrawl` step exercises the known fan-out amplification (~6-7x no-op behavior; one recrawl recently generated ~13k events) EVERY day, unattended, growing the log unboundedly. Two mitigations here: (a) the DEFAULT worklist is `advance_branch` — the cheapest useful step, NOT a recrawl — so out of the box the heartbeat never recrawls; `recrawl` is opt-in. (b) FOLLOW-UP, flagged explicitly: trigger-debouncing / fan-out reduction should land BEFORE `recrawl` is rotated into the unattended worklist, because the heartbeat compounds that cost daily. Trigger-debouncing is a change to the reactive fan-out core (its own invariant), too large to fold into this session safely; it is recorded as the next queued item gating unattended recrawl. Until it lands, the operator who adds `recrawl` to the worklist owns that cost knowingly.
- Residency: `lab_pack/heartbeat.py` is PLUMBING (droppable, like the watchdog). The three knobs (`heartbeat_cadence`, `heartbeat_worklist`, `heartbeat_budget_ceiling_usd`) are seam-eligible (`lab_pack/kernel.py` `SEAM_ELIGIBLE_SETTINGS`) so the operator tunes cadence/worklist/ceiling through the gate. The heartbeat's GATE-preservation and its clamp to the budget path stay in code, not in a seam the lab could move on itself.
- Fixture (`heartbeat`, LLM mocked — no live LLM): a tick within the window FIRES and advances exactly one worklist item, stamping its window; a second tick in the same window is a no-op; a simulated restart mid-window does not double-fire (the window key is read off the log); a new window fires again, advancing the next backlog item; spend over the ceiling records `heartbeat.skipped:budget` and does NO work; cadence="off" (an approved seam) disables it with no tick and no work; an operator worklist of "recrawl" rotates to the recrawl step and queues exactly one crawl_request; and across every leg the gate is unchanged — no decision approved/rejected, no submit_pr opened, nothing published, no steering verb fired. Suite green (40/40).
- Rationale: the lab should be able to make progress between operator visits without becoming an orchestrated system or weakening a single gate. A wall-clock tick that brings in one unit of fresh input, floors its own spend far below the cap, lands only proposals in a human inbox, and dies the instant the operator sets one seam to "off" is the smallest possible departure from reactive-only that buys continuous evidence growth — and it is recorded as a departure, not smuggled in as plumbing.
