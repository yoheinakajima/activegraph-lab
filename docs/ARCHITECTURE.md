# ARCHITECTURE

The lab is one layered pack (`lab_pack/`) on the activegraph-packs core pack, composed with upstream infrastructure packs in a bundle. CONTRACT.md governs. ADR-003/004/005/006 set the shape.

## The pack

`lab_pack/` follows the packs repo's `_template` layout: `__init__.py`, `object_types.py`, `behaviors.py`, `tools.py`, `settings.py`, `prompts/`, `fixtures/`, README, CHANGELOG. `requires = ["core"]`. `integrates_with = ["communication", "research", "codebase", "tool_gateway", "memory_gateway"]` — all optional, degrade gracefully. Consumed packs come from the pinned activegraph-packs git dependency; the lab pack is registered under the `activegraph.packs` entry point in this repo's pyproject.toml.

## Object types and relations

Three lab types: `mission` (title, statement, target_url, status), `branch` (title, intent, status, parent_branch_id, fork_event_id, authority), `decision` (subject_ref, kind, status, rationale, evidence_refs). All other nouns are core types. Relations: mission `has_branch` branch, branch `forked_from` branch, branch `produced` artifact, branch `supported_by` observation/evaluation, branch `dispatched` task, thread `discusses` branch (only when communication is loaded).

Relation call convention: this repo writes relations per the actual `Graph.add_relation(source_id, target_id, type)` signature (as the chat pack does), because runtime view traversal and relation queries match on the real source/target fields. The packs repo is split: core/research/tool_gateway write type-first, which the Inspector decodes by assuming inversion. A composed graph therefore contains both encodings; the lab's feed serializer discriminates per relation (object ids contain `#`, relation type names never do). Recorded as upstream friction per ADR-005.

## Behaviors

Twelve small reactive behaviors, no orchestrator:

- `ingest` — on mission.created or source-request events: fetch target_url and same-domain links through tool_gateway (registering a `fetch_url` capability if absent), create core sources, then observations for extracted claims. Depth ≤ 2, page cap ≤ 30, a progress event per page.
- `plan` — llm_behavior. On new observations under a mission: identify weakly evidenced claims, create proposed branch objects. Reasoning is narrated in the event payload, never scored by formula.
- `work` — on branch.status → active: create core tasks with routing tags (word-boundary matched: a claim DESCRIPTION mentioning "implements" is not a request to write code — the branch#64 misroute). A branch carrying an operator_direction (ADR-027) dispatches the latest direction verbatim on the task. If no pack reacts within a bounded window, record the capability gap as an observation — a gap is evidence, not an error, and no longer an exit: `blocked` is a task OUTCOME (the gap rides the blocked patch as result_summary), so interpret fires on it and a promote decision surfaces instead of the branch dangling silently.
- `interpret` — llm_behavior. On any task outcome under a lab branch (completion, failure, or blocked): write a summary observation, link evidence, set the branch to decided or propose follow-ups.
- `self_repair` — the self-repair loop's planner (ADR-036). On the lab's OWN defect observations (a `routing_miss` — ADR-031's verdict that a task was a routing/code bug, not a capability absence; a finding the build tagged `metadata.code_defect`): propose a GATED code-fix branch routed `codebase.code_task`, carrying the defect's evidence as the task intent. Deterministic (no LLM). Bounded: the lab's OWN allowlisted repo only (`yoheinakajima/activegraph-lab` — never the packs repo or core); concrete evidence only (a routing_miss is concrete by construction; a finding must be explicitly `code_defect`-tagged — free-text cue-matching would re-propose already-fixed bugs); gated like any branch (the operator activates); capped at `setting.max_self_repair_branches`; dark unless the code worker is live (no proposing into a dead lane). The downstream `submit_pr` the code worker opens stays doubly-gated + MCP-excluded; a fix that fails its sandbox proof opens none. The operator's `fix`/secondary front door is the `answer` `fix` steering verb.
- `gate` — on decision.created (pending): emit an approval-request event. Nothing publishes, self-modifies, or opens a pull request without an approved decision. A resolved `submit_pr` decision (ADR-035) is the doubly-gated PR last mile: approval is the ONE place the `GITHUB_WRITE_TOKEN` is exercised (push a branch, open a PR via `lab_pack/github_write.py` — token header-only, never in the graph), recording a `pr_opened` (or `pr_failed`) observation; rejection records the rationale (`pr_rejected`) and touches no token. submit_pr is EXCLUDED from MCP exactly like approve/reject — opening the PR is the operator's tap. The operator then reviews and merges on GitHub (the second gate); no auto-merge. No exceptions, including fixtures. An approved publish decision is the publishing last mile: artifact patched to status=published with published_at, slug uniqueness enforced (collision → numeric suffix), and an `artifact.published` marker event appended (ADR-013). A resolved promote decision lands the branch on `decided` either way (ADR-027): rejection records the operator's resolution_rationale as an operator_direction observation in the branch's evidence — teaching, never burial; nothing archives a branch anymore. An approved promote decision on a branch with ≥ research_min_evidence linked evidence objects yields ONE research draft_request; thinner decided branches wait until ≥2 of them can be synthesized over the same combined bar (ADR-014).
- `digest` — editorial accumulator (ADR-014). Finding-tagged observations are QUEUED (a queued_finding relation), never drafted directly; when unpublished queued findings reach digest_min_findings, one note-kind draft_request covers them all. Also captures per-observation provenance (actor + timestamp from the creation event) and feeds the branch-evidence registry from relation.created events — BehaviorGraph cannot iterate relations, so evidence counting inside behaviors is registry-backed, rebuilt from the graph on resume.
- `draft_writer` — llm_behavior. On draft_request observations ONLY (ADR-014): write a core artifact (kind=blog_draft, metadata.post_kind note|research|build, markdown with evidence footnotes, provenance block), mirror it to drafts/<slug>.md (graph copy canonical), and open a pending publish decision. Advisory review notes append to the body for operator attention, never auto-blocking: claims-coverage + process-claims + orphan-footnote (ADR-029), the overclaim lint (ADR-033: graph-grounded flags for "independent" over single-origin evidence, "autonomous" where an operator is in the chain, "verified"/"proven" over descriptive-only evidence, unfootnoted superlatives), and a one-line evidence profile (live vs inherited findings, distinct-branch span — supports the new-evidence-only bar). The request's data carries the code-injected draft context: classification guidance and, per item, the originating branch title, when and by what behavior the observation was created, and whether it was seeded or arose from live work — the model can no longer not know where a finding came from. An operator-requested draft carries the operator's FULL message as a delimited OPERATOR BRIEF block (metadata.operator_brief + the request text — the evt_13857 compression fix, mirroring the seam-truncation fix): when present, the brief governs scope and the listed findings are available evidence, not the mandatory skeleton; briefless digest requests are unchanged. OPEN: spec asked rejected drafts to become status `archived`, but the core artifact enum has no such value and core is not ours to change (ADR-005) — rejected drafts map to `rejected` plus a REJECTED header on the mirror file.
- `research_intake` + `research_worker` — the lab-local research worker (ADR-020, `lab_pack/research_worker.py`, plumbing, droppable). Intake claims tasks routed research.deep_research and gathers sources through tool_gateway (per-task cap `setting.research_fetch_cap`); the llm_behavior synthesizes with per-claim source attribution, writes observations + a branch-linked evaluation, and completes the task — or fails it with the error in the event. Dark by default (`research_worker_enabled`); the server boot enables it. Non-research routing still records capability gaps.
- `code_intake` + `code_worker` + `code_author` — the lab-local code worker (ADR-035/037, `lab_pack/code_worker.py`, plumbing, droppable — mirrors the research worker). Intake claims tasks routed `codebase.code_task` (the lane that had no reactor), resolves a code-task spec (repo, command, optional ref/fix diff, optional brief — from `task.metadata.code_task` or parsed from the intent/direction), runs it in the repo sandbox (`lab_pack/repo_sandbox.py`: clone an allowlisted repo, run the command; for a fix-task apply the diff and re-run to PROVE it), and writes the captured output as attributed evidence (a `code_run` observation linked to a repo source) + a code-synthesis request. AUTHORING (ADR-037, the last mile; the model contract amended by ADR-038 then ADR-042): a propose_pr task carrying a BRIEF but NO candidate diff diverts to `code_author` (an llm_behavior on `setting.model.code_worker` — code authoring is top-tier reasoning, routed via the `code_author → code_worker` model alias) which reads the relevant files from the clone (`repo_sandbox.clone_and_read` — a clean-env clone + bounded file read, no command run), emits per file search/replace EDITS to MODIFY an existing file or full CONTENT to CREATE a new file (NOT a unified diff) and adds/extends a regression test when the brief describes a defect; the lab MATERIALIZES the new contents (`repo_sandbox.materialize_authored_files` — apply each edit to the cloned original; a non-matching search block is an error fed to the retry) and BUILDS the patch deterministically (`repo_sandbox.build_diff_from_new_files` — difflib over the cloned original, git-apply-valid by construction, so no hand-computed `@@` hunk header can break `git apply` — the branch#1667 failure mode), applies it in the sandbox, and runs the proof command — the RUN decides success, never the model. Search/replace bounds the model's output to the CHANGE, so a large file no longer has to be re-emitted whole to fix a few lines (the branch#1837 max_tokens truncation). Pass → the existing propose_pr → gated submit_pr path with the git-BUILT diff + the proven post-fix file states; fail → the captured output is fed back for a bounded retry (`setting.code_author_max_attempts`, default 2), then an honest `authoring_unproven` evaluation ("authored a diff but could not make it pass") and NO PR — a fix earns its PR by proving in the sandbox. The synthesis llm_behavior (model `setting.model.code_worker`, default `claude-opus-4-8`) summarizes the run honestly and completes the task — done if the deciding run exited 0 (proven in the sandbox), failed otherwise; the verdict is the run's, computed by the lab, not the model's. Run cap `setting.code_run_cap`; per-run wall-clock `setting.sandbox_timeout_seconds`; the LLM stages ride the provider wrapper (pause + cost cap). Dark by default (`code_worker_enabled`); the server boot enables it. An explicit `code_task` spec routes codebase.code_task regardless of the intent's verb. Safety net (ADR-031 addendum): a read-to-verify task that lands in the codebase lane records a `routing_miss` naming research.deep_research (which carries get_file) and fails with an actionable verdict — branch#847 is answerable either way. New tags: `code_authoring_request`/`code_authored_run`/`code_authoring_failed` (the authoring stage).
- `seam_writer` — llm_behavior (Phase 4 rails). On seam_proposal_request observations — assembled by answer's "propose …" steering intent with the seam's current version verbatim, the operator's message IN FULL (never an excerpt — the decision#195 truncation), and the seam-relevant cited evidence (the operator's message always; rejected decisions only for prompt.draft_writer via publish rejections, or when a rejected decision references the same seam) — authors the next seam version through propose_seam_fn: artifact + PENDING self_modify decision whose evidence_refs record what informed it. Text the operator marked VERBATIM (`VERBATIM:` … `END VERBATIM`, or to end of message) is checked post-generation: if it does not appear intact in the body (substring after whitespace normalization), NO proposal opens — a seam_proposal_failed observation records the diff and a reply in the branch thread says so. Nothing auto-applies; the reserved prompt.draft_writer voice episode stays reserved.
- `answer` — llm_behavior, active only when communication is loaded. On message intents in threads that discuss a branch: answer from current graph state, stamp the event horizon, include provenance refs. Steering messages also write the corresponding object mutation. Steering verbs: pause/resume/activate/deactivate/draft/approve/reject/recrawl/note/propose `<seam>`, plus `fix` (ADR-036, the operator front door to the self-repair loop — proposes a code-fix branch from the message and activates it; "code" is deliberately not an alias, it is the noun ADR-025's routing avoids).

## Event taxonomy

The runtime owns the taxonomy: `object.created` / `object.patched` / relation events, plus `llm.requested`/`llm.responded` for llm_behaviors. The lab adds no STATE-BEARING event types; lab semantics (progress, approval-request, narration) are payloads on graph mutations — progress is an observation patch, an approval request is a pending decision object. Four marker events are appended but project no graph state: `artifact.published` (ADR-013), `lab.paused`/`lab.resumed` (ADR-015), and `lab.budget_set` (ADR-021) — pure log entries the lab's own projections and boot-time rebuilds read. The feed is a projection over these events joined with their objects.

## Worker coordination (ADR-006)

No adapters. `work` writes core tasks with routing tags (OPEN: exact tag convention — currently `task.metadata.routing` + tags in metadata). Verified against the packs repo at the current pin: no research or codebase behavior reacts to core task objects (research reacts to `source(kind=research_paper)`). The lab-local research worker (ADR-020) now fills the research.deep_research gap from inside the lab — implementing the same routing contract a future upstream pack would, and droppable the day one does. All other routing still produces capability-gap observations.

## Two-plane latency model

Two planes over one log. Fast plane: `answer` responds in one behavior cascade by reading current graph state, never waiting on tasks. Slow plane: `ingest`/`work`/`interpret` commit events at their own pace. The fast plane sees the slow plane only through committed events — hence the event-horizon stamp on every answer.

## Bundle and server

`lab_pack/bundle.py: build_lab()` composes core, tool_gateway, secrets, memory_gateway, agent_profile, identity_auth, communication, chat, research, codebase, and lab_pack, then creates the mission for https://activegraph.ai with the `read_the_website` seed branch. `server/` is a thin HTTP server copying the demo_server.py pattern: `/graph`, `/trace`, `/chat`, `/reset`, plus read-only `GET /lab/feed`. SQLite persistence under `data/`, paths overridable via env vars. No new storage, no new state.

## Seams (ADR-012, Phase 4)

Prompts, feed narration templates, and whitelisted settings are the lab's first self-modification surface. A seam is a core artifact (kind=seam, metadata: seam_name, version, parent_version) promoted through a decision (kind=self_modify) that the gate treats exactly like publish. `lab_pack/seams.py` (kernel) resolves each seam_name to the highest approved version with file fallback, hot-loads on approval (no restart), and refuses bodies referencing the kernel manifest at proposal AND load time. Inside behaviors the graph is restricted, so resolution is cache-only — populated exclusively by hot_load and boot-time apply_approved, which is also why a seam can never activate without passing the gate. Replay fidelity: behaviors stamp consumed seam versions onto their outputs (replay never re-fires behaviors, so outputs replay verbatim), and feed templates resolve as-of each entry's event via the approval events in the log.

## The charter (ADR-018)

The operator-authored mission charter is seam `charter.mission`: versioned, gated, hot-loaded, replay-recorded. v1 ships as the file default (`lab_pack/prompts/charter.md`; the file default counts as version 1 — unique among seams), and its body is injected verbatim as a delimited CHARTER block into the context assembly of plan, interpret, and draft_writer (answer is excluded by design). Behaviors stamp the charter version in force alongside their prompt versions; prompt seams and the charter compose, so promoting either recomposes the live context without restart.

## Graph code (ADR-012, Phase 5 — dark)

Behaviors and tools drafted as artifacts (kind=behavior_draft/tool_draft) with declared subscriptions, touched types, and authority. The pipeline — static checks (AST: no kernel-manifest imports, no os.environ, no network primitives outside tool_gateway, no exec/eval), sandbox subprocess run with a wall-clock timeout against a fake graph, declared-scope check — records each step as an evaluation event; only a full pass opens a pending self_modify decision. The loader (`lab_pack/graph_code.py`, kernel) activates approved drafts ONLY under LAB_ALLOW_GRAPH_CODE=1 (an approved decision alone is not enough); loaded behaviors run through a tagging graph proxy so every object they create carries draft provenance. The flag is intentionally absent from the deploy configuration: approved drafts list as dormant in the Seams view until the operator flips it.

## The repo sandbox, the code worker, and the self-repair loop (ADR-035)

GitHub rung 2 (ADR-022 rung 2, finally built): the rails that let a production bug become a PR the operator approves.

- **Repo sandbox** (`lab_pack/repo_sandbox.py`, plumbing) — a bounded execution sandbox built on the graph-code sandbox's isolation pattern (subprocess, wall-clock timeout, clean env — ADR-012 Phase 5), generalized to clone and run allowlisted repos. CLEAN ENVIRONMENT: the subprocess inherits no secret — its env is built from scratch from a curated allowlist, never a copy of `os.environ`, and `clean_env()` refuses to launch if a known secret key leaked in. WALL-CLOCK + RESOURCE LIMITS: killed at `setting.sandbox_timeout_seconds`, with POSIX CPU/AS/core rlimits best-effort. ALLOWLISTED REPOS ONLY (GITHUB_REPO_ALLOWLIST), checked before any I/O. CAPTURED OUTPUT (stdout/stderr/exit/timing) is evidence. Subprocess-hardened to start; **E2B is the documented future upgrade for running UNTRUSTED code** (a hosted micro-VM with its own kernel + network namespace) — this session takes no E2B dependency and runs only the lab's own allowlisted repos, where the clean-env subprocess + allowlist is the right floor and `run_repo_task` is the seam E2B would slot behind. The secret-isolation invariant is sentinel-gated: `scripts/test_sandbox_isolation.py` plants sentinel secrets in the parent env and asserts none is reachable inside the sandbox or in any captured output — a HARD GATE.
- **Code worker** (`lab_pack/code_worker.py`) — the reactor for `codebase.code_task` (see Behaviors above): clone → run → (for a fix-task) apply diff + re-run to prove → evidence + task-outcome so interpret fires. ADR-037 (the last mile; contract amended by ADR-038 then ADR-042): a propose_pr task with a BRIEF but NO candidate diff AUTHORS one — `code_author` reads the cloned files, the LLM (on the code_worker model tier) emits search/replace EDITS for an existing file (and full CONTENT for a new file like a regression test) — NOT a unified diff — bounding the output to the change; the lab MATERIALIZES the new contents and BUILDS the patch deterministically (difflib over the cloned original) and applies + proves it, retrying up to `setting.code_author_max_attempts` on failure (a non-matching search block is fed to the retry); a proven authored fix flows into the same propose_pr path carrying the git-built diff, an unprovable one records an honest gap and opens no PR. ADR-036: a PROVEN fix the lab was asked to LAND (`code_task.propose_pr`, set by the `self_repair` planner and the operator `fix` verb) opens a PENDING `submit_pr` carrying the diff + the post-fix file states the sandbox read back, citing the run's `sandbox_proven` evaluation — bounded to the lab's own repo at the PR mouth too (defense in depth). A fix that fails its proof opens none. ADR-039 (proof integrity): when the change authors a test, the proof runs THAT test directly (`run_repo_task(test_paths=…)` → `python -m pytest`) and the fix proves ONLY if the test is actually executed AND passes — a test-only no-op fails its own regression test against the unfixed clone and opens no PR (the branch#1704 false green); the authoring step reads the FULL target file (`clone_and_read` reads hint/target files uncut, the request renders them whole); a `pr_failed` carries a `failure_class` (credential wall vs other git/API error).
- **submit_pr decision** (kind=`submit_pr`, ADR-035) — a code change becomes a `code_change` artifact (diff for the reviewer + the concrete file contents to commit) plus the sandbox `sandbox_proven` evaluation, surfaced as a PENDING decision. ONLY operator approval exercises `GITHUB_WRITE_TOKEN` (a separate secret from the read `GITHUB_TOKEN`, fine-grained, lab-repo-only, absent until configured; read only in `lab_pack/github_write.py`, header-only, never in the graph/logs/payloads — sentinel-audited in `scripts/test_public_safety.py`). MCP-excluded like approve/reject (annotate is fine). No auto-merge — the operator merges on GitHub, the second gate.

The self-repair loop, end to end (documented in ROADMAP.md): production failure → finding/routing_miss observation → **the lab self-dispatches a fix** (ADR-036: the `self_repair` planner proposes a gated code_task branch from its own observed defect, no operator authoring) → the operator activates → **the code worker AUTHORS the diff** (ADR-037: from the brief + the cloned files, with a regression test) → the sandbox proves it → a PENDING submit_pr decision → operator approves → PR → operator merges → Replit pulls. ADR-035 built and proved the rails; ADR-036 closed the first mile (the proposal); ADR-037 closed the last mile (writing the fix). Still no auto-merge: every PR is the operator's tap. New marker/observation tags: `code_progress`/`code_run`/`code_synthesis_request`/`code_finding` and `self_repair_pr_proposed`/`self_repair_blocked` (the code worker), `pr_opened`/`pr_rejected`/`pr_failed` (the gate's submit_pr last mile), and `self_repair_capped` (the planner's concurrency cap). The bound on self-dispatch — own allowlisted repo only, capped, gated, dark unless the code lane is live — lives in CONTRACT.md (Workers).

## OPEN: prompt.draft_writer prose

The draft_writer prompt file is deliberately untouched by the Phase 4/5 editorial work — all new guidance (post_kind classification, per-finding provenance) is injected by prompt-assembly CODE into the draft_request observation the behavior reads. The first improvement to the prompt's PROSE is reserved for the lab's own first self_modify proposal through the seam gate, motivated by the rejected budget-tracking draft: the lab should argue for its own wording change with that rejection as evidence, and the approval should be a recorded decision, not a commit.
